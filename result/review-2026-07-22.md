## A. Stage 1 — advisor 백엔드 (FastAPI)

전반 견고. 요청 시점 코드가 DB를 안 타고(참조=파일, ODR2=사전집계 아티팩트) latest_run 윈도우·ODR2 결정성·seq 캐싱·에러봉투·CORS는 검증·수정 완료. 남은 결함:

### A-1 [High] ODR2 아티팩트 손상/권한오류 시 500 (503이어야 함)
- **위치**: [backend/app/routers/routes.py:26-30](../backend/app/routers/routes.py#L26-L30), [:40-46](../backend/app/routers/routes.py#L40-L46) — 원인 소스 [backend/app/queries/routes.py:18-27](../backend/app/queries/routes.py#L18-L27), [:30-40](../backend/app/queries/routes.py#L30-L40)
- **문제**: 두 엔드포인트가 `except FileNotFoundError`만 503으로 매핑한다. 하지만 `_load()`/`_load_meta()`는 파일이 잘리면 `json.JSONDecodeError`, 권한/IO 문제 시 `OSError`를 던져 잡히지 않고 전역 `Exception` 핸들러로 흘러 **500**이 된다. 게다가 `data_period()`(→`odr2_meta.json` 로드) 호출이 `try` 블록 **밖**(라인 30·46)이라, `odr2.json`이 멀쩡해도 `odr2_meta.json`만 깨지면 정상 경로가 500이 된다. `routers/reference.py`는 이미 `_ASSET_ERRORS = (FileNotFoundError, OSError, json.JSONDecodeError)`로 503 통일 패턴을 확립했는데 routes만 이탈 — docs/03 §7("자산 미가용=503")과도 불일치.
- **발생 시나리오**: 배치가 쓰다 죽어 `odr2.json`이 부분 JSON이거나, `odr2_meta.json`만 손상된 상태에서 `GET /api/routes/od-pairs` 또는 `GET /api/routes?dep=&arr=` 호출 → 503 대신 500.
- **수정방안**: routes 라우터에서 reference와 동일한 `_ASSET_ERRORS` 튜플을 잡아 503으로 통일하고, `data_period()` 호출을 `try` 안으로 이동(또는 예외 안전 래핑해 실패 시 `None`). 두 라우터가 공유 헬퍼로 같은 에러 매핑을 쓰게 하면 재발 방지.

### A-2 [Medium] 레이트리밋 `_hits` 딕셔너리 무한 증가 (메모리 고갈)
- **위치**: [backend/app/middleware.py:29](../backend/app/middleware.py#L29), [:31-44](../backend/app/middleware.py#L31-L44)
- **문제**: `_hits: dict[str, list[float]] = defaultdict(list)`에 요청마다 IP 키가 생기고, 만료 타임스탬프는 `pop(0)`로 정리하지만 **리스트가 비어도 IP 키 자체는 절대 삭제되지 않는다**. 인증 없는 공개 API라 소스 IP가 늘수록 딕셔너리가 무한 증가 → 레이트리밋이 막으려던 남용 시나리오에서 오히려 취약점.
- **발생 시나리오**: 다수 IP(스캐너/봇넷/프록시 미경유 다수 클라이언트) 장기 유입 시 프로세스 RSS 지속 증가.
- **수정방안**: 윈도우 정리 후 리스트가 비면 `del self._hits[client_ip]`, 또는 N요청/시간마다 `cutoff` 이전만 남은 빈 키 청소. 단일 인스턴스 전제는 유지하되 키 증가 상한만 둔다.

### A-3 [Low] bbox 경도 nan/inf 미검증 → 400 대신 조용한 빈 결과
- **위치**: [backend/app/reference/loader.py:42-54](../backend/app/reference/loader.py#L42-L54)
- **문제**: 위도는 `-90~90`·역전 검증하나 경도는 `min_lon>max_lon` 역전만 본다. `nan`/`inf` 경도는 `nan > nan == False`라 역전 검사를 통과하고 이후 비교가 전부 False가 돼 **빈 배열**을 반환한다(이 파일이 "조용한 빈 결과"를 없앤 취지와 모순). 경도 범위 clamp를 뺀 것(±360 연속좌표 허용)은 타당하나 유한성은 별개.
- **발생 시나리오**: `bbox=30,nan,40,200` → 원인 불명확한 빈 결과.
- **수정방안**: 경도 두 값에 `math.isfinite()` 검사 추가, 비유한이면 `ValueError`(→400). 범위 clamp는 그대로 두지 않음.

### A-4 [Low] `Query(..., regex=...)` deprecated (향후 업그레이드 시 깨짐)
- **위치**: [backend/app/routers/reference.py:83](../backend/app/routers/reference.py#L83)
- **문제**: `regex=r"^[A-D](,[A-D])*$"`는 현재 핀 버전(FastAPI 0.115/Pydantic 2.13)에선 DeprecationWarning으로 동작하나 상위 버전에서 제거되어 import 시점에 라우트가 깨진다.
- **수정방안**: `regex=`를 `pattern=`으로 교체(동작 동일, 경고 제거).

### A-5 [Low·선택] 모듈 전역 캐시 콜드스타트 스레드 경합 (양성)
- **위치**: [backend/app/reference/loader.py:23-28](../backend/app/reference/loader.py#L23-L28), [backend/app/queries/routes.py:18-27](../backend/app/queries/routes.py#L18-L27)
- **문제**: 동기 `def` 엔드포인트는 스레드풀 실행이라 콜드스타트 동시 요청 시 여러 스레드가 `if key not in _cache` 통과 후 동시 `json.load` 가능. GIL로 손상은 없고 결과 동일(중복 로드 1회) — 무해하나 순간 파싱 비용.
- **수정방안(선택)**: 로드 구간 `threading.Lock` 보호 또는 `functools.lru_cache` 대체. MVP 필수 아님.

## B. Stage 0 — 전처리 적재 백엔드

SQL은 전부 SQLAlchemy Core 파라미터/화이트리스트, 경로주입·업로드 크기·zip-bomb·hmac 토큰 비교·인코딩 폴백은 처리됨. 남은 결함:

### B-1 [High] 요청량 제한(rate limit)이 전혀 구현돼 있지 않음 — 문서는 "통과"로 기록
- **위치**: [data-ingestion-backend/app/main.py:12-19](../data-ingestion-backend/app/main.py#L12-L19)(미들웨어 0개) vs [docs/07-checklist.md:13](../docs/07-checklist.md#L13)(공통 게이트 "요청량 제한(rate limit)")·[:44](../docs/07-checklist.md#L44)("공통 게이트…통과"), [docs/06-conventions.md:91](../docs/06-conventions.md#L91)("조직 정책: API 자동화 시 요청량 제한·로그 기록 필수")
- **문제**: `main.py`에 CORS/rate-limit 등 어떤 미들웨어도 없고 `requirements.txt`에도 관련 의존성이 없다. Stage 1(advisor)은 `app/middleware.py`로 슬라이딩윈도우 레이트리밋을 실제 구현했는데, 같은 게이트를 "통과"라 적은 Stage 0에는 대응 코드가 없다. 무인증 `POST /uploads`·대용량 `download.csv`가 무제한이라 반복 호출로 subprocess/DB 부하 유발 가능. **조직 정책 위반 + 문서-코드 불일치(§D-1)**.
- **수정방안**: FastAPI 미들웨어(Stage 1 `app/middleware.py` 슬라이딩윈도우 패턴 재사용 권장)를 `main.py`에 등록, 한도는 하드코딩 금지 원칙대로 `config.py`에 `RATE_LIMIT_PER_MINUTE` 등 env 추가. 요청 로깅 미들웨어도 함께. 리버스 프록시 전제로 갈 경우 그 사실·설정을 `docs/08`에 명시하고 checklist 문구 정정.

### B-2 [Medium] idempotency 경합(IntegrityError→기존 run 리다이렉트) 경로에서 워크스페이스 디렉터리 누수
- **위치**: [data-ingestion-backend/app/routers/upload.py:126-133](../data-ingestion-backend/app/routers/upload.py#L126-L133)(특히 :132 `RedirectResponse`), 정리 로직 [:140-143](../data-ingestion-backend/app/routers/upload.py#L140-L143)(except)·[:144-145](../data-ingestion-backend/app/routers/upload.py#L144-L145)(finally)
- **문제**: `ws`(격리 워크스페이스, flight_data는 공간데이터 전체 copytree된 큰 디렉터리)는 `except`에서만 `rmtree(ws)`로 정리된다. create_run의 IntegrityError를 잡아 기존 run으로 `return RedirectResponse`하는 경로는 예외를 안 던지므로 `except` 미실행, `finally`는 `tmp_dir`만 지운다 → **`ws`가 디스크에 영구 잔류**.
- **발생 시나리오**: 동일 `idempotency_key` 두 요청이 초기 조회를 동시 통과 → 둘 다 create_run → 한쪽 unique 위반 IntegrityError → 리다이렉트하며 방금 만든 `ws`(+복사 공간데이터)가 orphan. 반복 시 디스크 누적.
- **수정방안**: IntegrityError 분기의 리다이렉트 직전에 `if ws is not None: shutil.rmtree(ws, ignore_errors=True)`. 또는 정리를 플래그 기반 `finally`로 옮겨 조기 반환 경로 전부 커버.

### B-3 [Medium] 크래시로 QUEUED/RUNNING에 고착된 run을 복구·삭제할 방법이 없음
- **위치**: [data-ingestion-backend/app/ingestion/pipeline.py:32-47](../data-ingestion-backend/app/ingestion/pipeline.py#L32-L47)(in-memory `BackgroundTasks` 실행), [app/ingestion/loaders.py:472](../data-ingestion-backend/app/ingestion/loaders.py#L472)(`_DELETABLE_STATUSES`=terminal만), 삭제 UI 게이트 [app/templates/run_detail.html:28](../data-ingestion-backend/app/templates/run_detail.html#L28)
- **문제**: run은 QUEUED 생성 후 프로세스 메모리 `BackgroundTasks`로만 전이. 배경작업 전 프로세스가 죽으면 QUEUED로, RUNNING 중 죽으면 RUNNING으로 영구 고착. 그런데 `delete_run`은 terminal 상태만 허용(loaders.py:494)해 정리 불가, 재시작 시 정리/재개하는 리퍼도 없다.
- **발생 시나리오**: 대용량 처리 중 uvicorn 재시작/OOM → run이 RUNNING 영구 잔류, 목록에 "처리중"으로 남고 삭제 불가.
- **수정방안**: (a) startup 훅에서 이전 프로세스가 남긴 QUEUED/RUNNING을 FAILED(`error_code="INTERRUPTED"`)로 정리 또는 재큐잉, 그리고/또는 (b) 관리자 토큰 하에 `delete_run`이 RUNNING/QUEUED 강제 정리 경로 제공. 최소한 문서에 한계 명시하고 checklist "완료" 문구 정정.

### B-4 [Low] `start.sh`가 운영에서 부적절한 `--reload`로 기동
- **위치**: [data-ingestion-backend/start.sh:30](../data-ingestion-backend/start.sh#L30)
- **문제**: `--reload`는 개발용. 이 앱은 프로세스 전역 `threading.Lock`(pipeline.py:29) 단일 워커 전제라, reloader 재시작 중 in-memory BackgroundTasks가 유실될 수 있다(B-3과 연동).
- **수정방안**: `INGESTION_RELOAD` env 토글(기본 off)로 개발/운영 분리, `--host`도 env화.

### B-5 [Low] `loaders.has_running_run()`는 정의됐으나 미사용(죽은 코드)
- **위치**: [data-ingestion-backend/app/ingestion/loaders.py:100-104](../data-ingestion-backend/app/ingestion/loaders.py#L100-L104)
- **문제**: 순차 가드는 `pipeline._PIPELINE_LOCK`(스레드 락)으로만 이뤄지고 DB 기반 이 함수는 미사용 — 다중 워커/재시작 시 무엇이 진실인지 혼동.
- **수정방안**: 제거하거나 다중 워커 대비 실제 가드로 채택할지 결정해 통일, 근거를 `기술스택_결정.md §2.6`에 반영.

## C. Stage 2 — 프론트 (Leaflet)

F1~F10 구현·escapeHtml 전면 적용·에러봉투 파싱·selectOd 세대가드는 확인됨. `config.example.json`↔`DEFAULT_CONFIG` 동기화(F1)도 일치 확인. 남은 결함:

### C-1 [High] `setViewMode` 경쟁조건 — 세대 토큰 가드 없음
- **위치**: [frontend/js/store.js:152-165](../frontend/js/store.js#L152-L165) (`selectOd`엔 `odRequestSeq` 가드가 있으나 [:97](../frontend/js/store.js#L97)~[:122](../frontend/js/store.js#L122)) `setViewMode`엔 대응 가드 없음), [frontend/js/viewmode.js:24-39](../frontend/js/viewmode.js#L24-L39)(로딩 중 버튼 비활성화도 없음)
- **문제**: `setViewMode(mode)`가 `ensureBulkLoaded()`를 await한 뒤 무조건 `state.viewMode = mode`로 덮어쓴다. 두 번의 전환이 겹치면 늦게 끝난 이전 요청이 최신 선택을 덮어써 상태/UI 불일치. `viewmode.js`도 `setActiveButton(mode)`를 await 뒤에 호출해 버튼 강조까지 경쟁.
- **발생 시나리오**: `region` 클릭(벌크 3초 로드) → 도중 `focus` 클릭(focus는 로드 없이 즉시 반영) → 이후 region 로드 완료가 `state.viewMode="region"`으로 덮어씀 → 사용자는 focus를 눌렀는데 화면은 region. (재전환하면 복구되나 명백한 논리 버그.)
- **수정방안**: `selectOd`와 동일한 세대 토큰 도입:
  ```js
  let viewModeSeq = 0;
  export async function setViewMode(mode) {
    const seq = ++viewModeSeq;
    notify({ type: "viewmode:changing", mode });
    try {
      if (mode !== "focus") await ensureBulkLoaded();
      if (seq !== viewModeSeq) return;      // 더 최신 전환이 진행 중 — 폐기
      state.viewMode = mode;
      notify({ type: "viewmode:changed", mode });
    } catch (err) {
      if (seq !== viewModeSeq) return;
      notify({ type: "viewmode:error", message: describeError(err) });
      throw err;
    }
  }
  ```
  추가로 `viewmode.js`에서 전환 진행 중 토글 버튼 `disabled` 처리(중복 클릭 자체 억제)를 권장.

### C-2 [Medium] `route.js` FIR 하이라이트가 04-F 겹침 폴리곤 수정(normalizeWinding)을 누락
- **위치**: [frontend/js/layers/route.js:33-46](../frontend/js/layers/route.js#L33-L46) (`highlightFirs`) vs 적용된 곳 [frontend/js/layers/reference.js:33-35](../frontend/js/layers/reference.js#L33-L35)(`normalizeWinding`)·[:60](../frontend/js/layers/reference.js#L60)
- **문제**: `reference.js`는 04-F 버그(Canvas 기본 evenodd가 겹치는 멀티폴리곤을 상쇄)를 `fillRule:'nonzero'` **+ shoelace 링 방향 통일(`normalizeWinding`)**로 고쳤다. 그런데 `route.js`의 오렌지 FIR 하이라이트는 `fillRule:'nonzero'`만 쓰고 `normalizeWinding`을 적용하지 않는다. mixed-winding FIR(SBAO/SBBS/UEEE/UWWW 등 reference.js 주석이 실재 확인한 케이스)를 강조하면 채움이 상쇄돼 오렌지 면이 비거나 뚫려 보일 수 있다.
- **발생 시나리오**: 선택 경로의 경유 FIR에 mixed-winding FIR가 포함될 때 하이라이트 채움 결손.
- **수정방안**: `highlightFirs`에서 각 링을 `normalizeWinding`으로 정규화 후 넘긴다. `reference.js`의 `signedArea`/`normalizeWinding`를 공용 모듈(예: `js/geo.js`)로 추출해 두 레이어가 공유하면 재발 방지:
  ```js
  const rings = fir.polygons.map((ring) => [normalizeWinding(ring)]);
  ```

### C-3 [Low] ADS-B 폴링 타이머가 페이지 unload 시 정리되지 않음
- **위치**: [frontend/js/main.js:145](../frontend/js/main.js#L145)(`adsb.start()` 호출), stop은 있으나 미호출 [frontend/js/layers/adsb.js:151-154](../frontend/js/layers/adsb.js#L151-L154)
- **문제**: `stop()`은 구현돼 있으나 어디서도 호출되지 않아 `setInterval`이 unload까지 유지된다. 정적 단일 페이지라 탭 닫으면 자동 정리되지만 방어적 정리 부재.
- **수정방안**: `main.js`에서 `window.addEventListener("beforeunload", () => adsb.stop())` 추가(향후 SPA 라우팅 전환 대비).

### C-4 [Low·선택] 폴백 프록시 성공 경로가 리로드 간 유지되지 않음
- **위치**: [frontend/js/net.js:8-40](../frontend/js/net.js#L8-L40)
- **현황**: `createFallbackFetcher`는 인스턴스 클로저 `successPath`로 **세션 내(메모리) 성공 경로를 기억**하며, `weather.js`·`adsb.js`가 각각 모듈레벨 fetcher를 1개씩 써서 공항 A→B METAR 조회 간에도 경로를 공유한다(모듈 간 분리는 호스트별 CORS 차이 때문에 의도적, 파일 주석에 명시). 즉 "공항마다 direct→local→public을 처음부터 다시 시도"하지는 않는다.

## E. 확인된 오탐/의도적 설계 (수정 불필요 — 재작업 방지 기록)
