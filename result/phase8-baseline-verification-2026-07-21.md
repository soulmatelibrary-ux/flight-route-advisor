## 1. 기준선(ground truth) 생성 (§10.1)

문서 지침대로 4종 모두 **스킬을 직접 CLI로 재실행**해 기준선을 새로 만들었다(기존 `outputs/`를 그대로 재사용하지 않음 — ACDM은 특히 문서가 기존 `outputs/acdm_timeline_fixed`를 "검토필요 139행 이슈"로 금지).

- `--workspace`로 `SOURCE_PROJECT_ROOT`(원본 프로젝트 루트)를 그대로 지정해 실제 원본 폴더(`ACDM/`·`FOIS/`·`비행자료/`·`흐름관리일지/`·`공간데이터/`) 전체를 입력으로 사용
- ACDM은 4개 공항(인천·김포·김해·제주) × 출발/도착 8개 폴더 전부 포함
- flow-management는 Stage 1단계 정책과 동일하게 `--no-integrate` 사용

### 발견한 이슈 — macOS 파일명 유니코드 정규화(NFD) 문제

원본 폴더를 **직접** 워크스페이스로 지정해 스킬을 돌리자 `preprocess-flight-data`가 "분석 원본 파일 없음" 오류로 실패했다. 원인 확인: macOS 파일시스템은 한글 파일명을 NFD(분해형)로 저장하는데, 스킬 코드의 정규식 리터럴(`ANALYSIS_RE` 등)은 NFC(조합형)라 `fullmatch`가 실패한다. 실제로 바이트 비교로 확인함(`unicodedata.is_normalized`).

- 이 문제는 Stage 0 백엔드(`workspace_builder.py`)에는 **영향이 없다** — flight_data 파일명은 서술자 템플릿(`{prefix}_{date}{ext}`, Python 리터럴이라 NFC)으로 재구성되고, ACDM/FOIS/흐름관리는 파일명 내용을 정규식으로 매칭하지 않아(폴더명·날짜 접미사만 사용) NFD 여부와 무관하다. 실제로 이번 검증에서 백엔드 처리 결과는 전부 정상이었다.
- 기준선 생성(스킬을 원본 폴더에 **직접** 실행)에서만 문제가 됐으므로, 원본은 건드리지 않고 파일명만 NFC로 정규화한 임시 복사본(스크래치 디렉터리)을 만들어 그 위에서 기준선을 생성했다(내용은 원본과 바이트 단위로 동일, 파일명 유니코드 정규화만 다름).
- 이 복사본이 원인이 되어 흐름관리일지 처리결과의 "원본파일" 컬럼이 기준선(NFC)과 백엔드 실결과(원본 그대로, NFD) 사이에서 육안상 동일하지만 바이트가 달라 최초 비교에서 전 행 불일치로 나타났다 — 비교 스크립트에서 NFC 정규화 후 비교하도록 수정해 해소(실제 데이터 손상 아님).

## 3. 동일성 비교 (§10.2 ②③④)

pandas 스크립트(CSV/DB 전체를 직접 열지 않고 짧은 비교 스크립트만 사용, CLAUDE.md §6 준수)로 두 단계 비교:

1. **DB 적재값 ↔ 백엔드가 적재에 쓴 CSV** — 로더가 값을 훼손하지 않았는지
2. **백엔드 CSV ↔ 독립 기준선 CSV** — 웹 업로드→workspace 재구성→subprocess 경로가 직접 실행과 동일 결과를 내는지

문서 지정 자연키(비행자료: 고유ID, ACDM: operation_date+airport_icao+flight_icao, FOIS: 편명+출발/도착일자, 흐름관리: 흐름관리ID) 기준으로 비교하되, **자연키에 중복이 있는 행(예: 고유ID 중복 48건)에서 동점 정렬 순서가 갈려 거짓 불일치가 나는 것을 막기 위해 전체 컬럼 값으로 정렬해 멀티셋을 비교**했다(자연키만 정렬키로 쓰면 회전문 형태의 안정정렬이라도 DB SELECT 순서와 CSV 파일 순서 자체가 달라 동점 그룹 내 순서가 어긋날 수 있음 — 실측으로 확인).

결과: 6개 테이블 전부 행수 정확히 일치 + 전체 컬럼 값 완전 일치. 제외행(ACDM `검토필요` 등) 건수도 백엔드/기준선 양쪽 처리결과 행수가 정확히 같으므로(예: ACDM 출발 37,508행 = 입력 37,647행 - 제외 139행, 양쪽 동일) 간접적으로 일치가 확인됨.
