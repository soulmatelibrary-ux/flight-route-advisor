## 2. 선정 근거

### 2.1 웹 프레임워크 — FastAPI

사용자가 기존에 운영 중인 다른 프로젝트(myStock 등)에서 이미 FastAPI + uvicorn 조합에 익숙하다. 추가로:

- `UploadFile`로 멀티파트 파일 업로드가 표준 지원되어 업로드 폼 구현이 단순하다.
- `BackgroundTasks`가 프레임워크에 내장되어 있어, 별도 인프라 없이 "업로드 즉시 트리거되는 배치 처리" 요구사항을 바로 구현할 수 있다.
- 향후 API 자동 수신으로 확장할 때도 동일 프레임워크에서 REST 엔드포인트만 추가하면 되므로 재작업이 적다.

### 2.2 템플릿 — Jinja2 (서버 렌더링)

1단계 UI 범위는 "업로드 폼 페이지"와 "처리결과/로그 조회 페이지" 두 개뿐이다. SPA 프레임워크(React 등)를 도입하면 빌드 파이프라인, 별도 배포 대상, 상태관리 등 이 범위에 필요 없는 복잡도가 추가된다. 사용자가 기존 프로젝트에서 이미 Jinja2 + `Jinja2Templates` 패턴을 쓰고 있어 동일 관례를 재사용한다.

### 2.3 DB 계층 — SQLAlchemy 2.0 Core (풀 ORM 아님)

사용자는 지금까지 관계형 DB를 SQLite + raw SQL로 직접 다뤄왔고, ORM·PostgreSQL 경험이 없다. 이번이 두 가지를 동시에 처음 도입하는 지점이므로 학습곡선을 하나로 줄이는 선택이 필요하다.

- **SQLAlchemy Core**는 `Table`을 컬럼 단위로 명시하고 `insert()/select()/update()`를 호출하는 방식으로, 실행되는 SQL과 코드가 1:1로 대응되어 raw SQL 사고방식에서 가장 매끄럽게 전이된다.
- 선언형 ORM(관계매핑, lazy loading, 세션 identity map 등)은 이번 단계(단순 적재/조회, 객체 그래프 탐색 불필요)에는 과도한 개념이다.
- Core로 시작해도 Alembic autogenerate와 완전히 호환되므로, 향후 필요해지면 ORM으로 전환하더라도 마이그레이션 이력이 끊기지 않는다.

### 2.4 마이그레이션 — Alembic

PostgreSQL 생태계의 사실상 표준이며, 마이그레이션 스크립트가 순수 Python(구조는 SQL을 그대로 옮긴 형태)이라 raw SQL에 익숙한 사용자도 읽기 쉽다. **가장 중요한 이유는 이관 요구사항**이다: Supabase도 표준 PostgreSQL이므로, `DATABASE_URL`만 Supabase 커넥션 문자열로 바꿔 `alembic upgrade head`를 그대로 재실행하면 스키마를 이전할 수 있다. 수동 SQL 스크립트 관리 방식은 이 재현성을 보장하기 어렵다.

### 2.5 DB 드라이버 — psycopg2-binary (동기)

업로드가 수동·저빈도(1인 운영, 일/시간 단위)이므로 비동기 드라이버가 주는 동시성 이점이 크지 않다. SQLAlchemy Core와 가장 널리 검증된 동기 조합을 선택해 안정성을 우선한다. 향후 동시 사용자가 늘어 비동기가 필요해지면 `session.py` 한 곳에서만 드라이버를 교체하도록 DB 접근을 캡슐화한다(아래 4.3 참고).

### 2.6 백그라운드 처리 — `BackgroundTasks` (Celery/Redis 불필요)

다음 세 가지 근거로 Celery/RQ 같은 별도 작업 큐 인프라를 이번 단계에 도입하지 않는다.

1. 요구사항 자체가 "진짜 실시간 스트리밍이 아니라 파일 업로드 즉시 트리거되는 배치 처리"다.
2. 처리 대상 규모(기존 `outputs/` 산출물 기준 비행자료 8만여 행, ACDM/FOIS 수천~수만 행)는 스킬 실행 기준 초~분 단위로 끝난다.
3. Starlette의 `BackgroundTasks.add_task`는 동기 함수(예: `subprocess.run`처럼 블로킹되는 호출)를 넘겨도 내부적으로 스레드풀에서 실행하므로, 이벤트 루프를 막지 않고 로그 조회 페이지는 처리 중에도 계속 응답할 수 있다.

Celery/Redis는 브로커, 워커 프로세스, 모니터링이라는 운영 요소를 추가로 얹는데, 이는 Postgres+ORM+Docker를 동시에 처음 배우는 시점에 불필요한 복잡도다. **다만 동시에 2개 이상의 run이 겹치지 않도록, 앱 차원에서 "이미 RUNNING인 run이 있으면 새 업로드는 QUEUED로 두고 순차 처리"하는 간단한 가드가 필요하다.** 향후 API 자동수신으로 업로드 빈도가 급증하면 이 지점을 Celery/RQ로 교체하도록, "트리거"와 "실제 처리 로직"을 처음부터 분리해서 설계한다(작업계획서 3장 참고).

### 2.7 컨테이너 — Docker Compose 2서비스

`app`(FastAPI, `python:3.12-slim`, non-root `USER app`, `PYTHONDONTWRITEBYTECODE=1`/`PYTHONUNBUFFERED=1`)과 `db`(`postgres:16-alpine`, named volume)로 구성한다. 이는 사용자의 기존 프로젝트(myStock)의 Dockerfile/compose 관례를 그대로 재사용하는 것으로, 새로 배워야 할 것을 최소화한다. `db` 서비스는 로컬 전용이며, Supabase로 이전할 때는 이 서비스를 제거하고 `DATABASE_URL` 환경변수만 Supabase 커넥션 문자열로 교체한다.

### 2.8 설정관리 — `python-dotenv` + `dataclass Settings`

기존 프로젝트(myStock)와 동일한 관례를 유지해 새 개념(pydantic-settings 등) 추가를 피한다.

### 2.9 기존 전처리 로직 연동 — subprocess로 CLI 그대로 호출

기존 스킬 4개(`preprocess-flight-data`, `preprocess-acdm`, `preprocess-fois`, `preprocess-flow-management`)는 모두 `python <script> --workspace <path>` 형태의 CLI이며, `argparse` 기반으로 `if __name__ == "__main__":` 아래에서 실행되는 구조다(함수형 API가 아님). ACDM 스킬은 이미 내부적으로 `subprocess.run`을 통한 스테이지 체이닝과 uuid 기반 임시폴더 격리 패턴을 쓰고 있다.

| 방식 | 장점 | 단점 |
|---|---|---|
| **subprocess로 CLI 그대로 호출 (채택)** | 스킬 코드를 1바이트도 수정하지 않고 재사용 — "기존 로직 재사용, 재구현 금지" 원칙에 정확히 부합. 종료코드로 성공/실패가 명확. 스킬이 독립 zip으로 배포되는 구조와 결합도가 낮아, 스킬이 업데이트돼도 백엔드 코드를 거의 건드릴 필요가 없음. | 프로세스 기동 오버헤드(수백ms, 이 규모에서는 무시할 수준). 상태를 stdout/stderr 파싱으로 얻어야 함. |
| 스크립트를 함수로 import | 오버헤드 없음, 예외를 파이썬에서 직접 캐치 | 스킬 내부를 함수형으로 리팩터링해야 하므로 "새로 만들지 않는다"는 원칙과 충돌. 백엔드와 스킬 배포판이 강하게 결합돼 스킬 업데이트 시 백엔드도 깨질 위험. |

**결론: subprocess 호출을 표준으로 채택.** 백엔드는 각 스킬을 "블랙박스 CLI 도구"로 취급하고 (1) 종료 코드, (2) stdout/stderr, (3) 스킬이 만든 CSV/JSON 산출물 경로만으로 결과를 판단한다.

## 3. 명시적으로 채택하지 않는 것 (이번 단계 범위 아님)

- Celery/RQ, Redis, 메시지 브로커
- SPA 프론트엔드 프레임워크(React, Vue 등)
- 선언형 ORM 관계매핑, GraphQL
- CI/CD 파이프라인 구축(GitHub Actions 등) — Git 친화적 구조(`.gitignore`, 환경변수 분리)까지만 이번 단계에서 준비하고 실제 CI 구성은 이후 단계로 미룬다.
- Supabase 계정/프로젝트 생성 및 실제 이전 작업 — 이관이 쉬운 "설계"만 이번 단계에서 반영한다.

## 4. 향후 확장 시 교체 지점 (설계에 미리 반영)

이 항목들은 지금 구현하지 않지만, 나중에 바꾸기 쉽도록 지금부터 코드 경계를 분리해 둔다.

1. **업로드 빈도 급증 → BackgroundTasks에서 Celery/RQ로 교체**: "트리거(업로드 접수)"와 "실행(스킬 실행+적재)"을 별도 함수/모듈로 분리해두면, 나중에 트리거 지점만 큐 발행으로 바꿀 수 있다.
2. **동기 psycopg2 → 비동기 드라이버**: DB 접근을 `session.py` 한 곳으로 캡슐화해, 드라이버 교체가 그 파일에만 영향을 주게 한다.
3. **로컬 파일 저장 → Supabase Storage**: 원본 파일 저장 경로 로직(`workspace_builder`/`raw_files.stored_relpath`)을 인터페이스 뒤에 숨겨, 저장소를 교체해도 상위 로직이 바뀌지 않게 한다.
4. **수동 업로드 → API 자동 수신**: 업로드 라우터가 받는 입력(파일+메타데이터)과 그 이후의 처리 파이프라인을 분리해두면, API 엔드포인트를 새로 추가하는 것만으로 자동 수신을 지원할 수 있다.

## 5. 리뷰 반영 개선사항 (v1.1)

문서 리뷰 과정에서 "초기 구현 시 누락되기 쉬운 안전장치"를 다음처럼 확정한다.

1. **업로드 입력 검증을 필수화**
	- 허용 확장자(`.csv`, `.xlsx`) 화이트리스트 적용
	- 콘텐츠 길이 기반 최대 업로드 크기 제한(예: 기본 200MB, 환경변수로 조정)
	- 파일명은 저장 시 서버 생성명으로 치환하고 원본 이름은 메타(`raw_files.original_filename`)에만 기록(경로주입 방지)
2. **처리 멱등성 확보**
	- 동일 업로드 재시도 시 중복 run 생성 여부를 제어하기 위해 요청 단위 `idempotency_key`를 지원
	- 같은 키로 이미 `RUNNING/SUCCESS`인 run이 있으면 새 실행 대신 기존 run을 반환
3. **상태 전이 규칙 고정**
	- 허용 전이: `QUEUED -> RUNNING -> SUCCESS|VALIDATION_FAILED|FAILED`
	- 종료 상태(`SUCCESS`, `VALIDATION_FAILED`, `FAILED`)에서 재전이는 금지
4. **트랜잭션 경계 분리**
	- `raw` 적재 트랜잭션과 `processed` 적재 트랜잭션을 분리해, 후반 실패가 원본 추적성까지 롤백하지 않게 설계
5. **관측성 최소 기준 추가**
	- 모든 run에 대해 `run_id`, `run_type`, `started_at`, `finished_at`, `duration_ms`, `status`, `error_code`를 구조화 로그로 남김
	- 로그 조회 페이지에서 `stderr` 원문과 검증 요약 JSON을 함께 노출

## 6. 확정 의존성 목록 (backend/requirements.txt)

백엔드 컨테이너에는 **백엔드 자체 의존성 + 스킬 실행 의존성**을 함께 설치한다. 스킬을 subprocess로 호출하려면 스킬이 요구하는 패키지가 동일 파이썬 환경에 있어야 하기 때문이다(스킬은 미설치 시 설치 시도 없이 실패). 버전은 재현성을 위해 고정한다.

```text
# --- 백엔드 ---
fastapi==0.115.*
uvicorn[standard]==0.32.*
jinja2==3.1.*
python-multipart==0.0.*      # UploadFile 멀티파트 처리
sqlalchemy==2.0.*
alembic==1.14.*
psycopg2-binary==2.9.*
python-dotenv==1.0.*

# --- 스킬 실행 의존성 (00_시작안내/requirements.txt와 동일 버전 고정) ---
pandas==2.3.3
openpyxl==3.1.5
shapely==2.1.2               # flight-data 출발 FIR 보정에 필요
```

> `pandas`/`openpyxl`/`shapely` 버전은 원본 `00_시작안내/requirements.txt` 기준이다. 스킬 재현성이 이 버전에 의존하므로 임의로 올리지 않는다. 실제 배포 전 원본 requirements와 대조해 최신 고정값을 확인한다.

## 7. 환경변수 (.env.example)

`config.py`의 `Settings`가 읽는 값. 코드에 리터럴로 박지 않는다.

```dotenv
# DB (로컬 Docker)
DATABASE_URL=postgresql+psycopg2://app:app@db:5432/ingestion
# Supabase 이전 시: postgresql+psycopg2://<user>:<pw>@<host>:5432/postgres?sslmode=require

# 경로
REPO_ROOT=/app/repo                 # 스킬/원본 폴더가 있는 프로젝트 루트(컨테이너 내 마운트 경로)
UPLOAD_DIR=/app/data/uploads        # 원본 아카이브
WORKSPACE_ROOT=/app/data/workspace  # run별 임시 workspace

# 업로드 제한
MAX_UPLOAD_MB=200
ALLOWED_EXTENSIONS=.csv,.xlsx
```

> `REPO_ROOT`는 이 프로젝트 루트(`데이터전처리기술이식`)를 컨테이너에 마운트한 경로다. 스킬 코드(`skills/`)와 원본 데이터 폴더가 여기 있어야 하며, 컨테이너는 이를 **읽기 전용**으로 마운트하는 것을 권장한다(원본 불변 원칙). 단, flight-data가 공간데이터를 workspace로 복사하는 동작은 read-only 마운트에서도 문제없다(복사 대상은 쓰기 가능한 `WORKSPACE_ROOT`).
