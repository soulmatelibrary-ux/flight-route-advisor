# 스킬 연동 레퍼런스 (subprocess 호출 계약)

- 문서 버전: 1.0
- 작성일: 2026-07-21
- 목적: 백엔드가 기존 전처리 스킬 4종을 subprocess로 호출·검증·결과수집할 때 필요한 **정확한 인터페이스 사실**을 한곳에 모은다. 이 문서는 실제 스크립트 코드에서 검증한 것이며, SKILL.md와 코드가 충돌하면 **코드가 기준**이다.
- 관련 문서: [작업계획서](./작업계획서.md), [DB스키마](./DB스키마.md), [기술스택_결정](./기술스택_결정.md), [체크리스트](./체크리스트.md)

> 모든 파싱·전처리 로직은 이 프로젝트 루트의 `skills/` 폴더를 그대로 호출한다(재구현 금지). 원본 데이터는 각 원본 폴더(`비행자료/`, `ACDM/`, `FOIS/`, `흐름관리일지/`, `공간데이터/`)에서 읽는다. 스킬 스크립트(`skills/*/scripts/*.py`)는 **절대 수정하지 않는다.**

## 0. 경로 표기 규칙

- `<REPO>` = 이 프로젝트 루트 = `/Users/sein/Desktop/project/데이터전처리기술이식`
- `<WS>` = run별로 백엔드가 만드는 격리 임시 workspace 루트 (예: `data/workspace/<run_id>/`)
- 스킬 스크립트 자체는 `<REPO>/skills/...`에 있는 원본을 그대로 호출하되, `--workspace`만 `<WS>`로 넘긴다. 즉 **코드는 원본 위치, 데이터는 격리 workspace**로 분리한다.

## 1. 4종 공통 사실 (반드시 숙지)

1. **기간(날짜범위)은 CLI 인자가 아니다.** `--start`, `--end`, `--period` 같은 인자는 어느 스킬에도 없다. 기간은 입력 데이터/파일명에서 자동 산출되어 출력 파일명에 삽입된다.
2. **출력 파일명을 예측하지 마라.** 4종 모두 동일 파일명이 이미 있으면 `_2`, `_3` … 번호를 자동으로 붙인다. 백엔드는 "예상 파일명"을 가정하지 말고 **스킬이 stdout으로 반환한 실제 경로**를 파싱해서 사용해야 한다(파싱 방법은 §3의 각 스킬 항목).
3. **날짜 구분자가 스킬마다 다르다.** flight-data만 `YYYYMMDD_YYYYMMDD`(언더스코어), acdm/fois/flow는 `YYYYMMDD-YYYYMMDD`(하이픈).
4. **최종 CSV 인코딩은 UTF-8 BOM(`utf-8-sig`).** 단, fois/flow가 만드는 검증 JSON은 BOM 없는 UTF-8.
5. **필수 인자 차이**: acdm·fois·flow-management는 `--workspace`가 `required=True`. flight-data는 기본값이 `cwd`지만 **백엔드는 4종 모두 `--workspace`를 명시적으로 전달**한다.
6. **실행 환경변수**: 호출 시 `PYTHONUTF8=1`, `PYTHONIOENCODING=utf-8`를 주입한다(한글 경로/출력 안정성). ACDM은 내부 스테이지에서 이미 이 값을 주입하므로 일관성을 위해 4종 모두 동일하게 설정한다.
7. **파이썬 실행기**: `sys.executable`(백엔드 컨테이너의 파이썬)로 호출한다. 스킬 의존성(`pandas`, `openpyxl`, `shapely`)이 이 파이썬 환경에 설치되어 있어야 한다(스킬은 미설치 시 설치 시도 없이 실패). 버전은 [기술스택_결정](./기술스택_결정.md) 참고.
8. **실패 방식**: 4종 모두 실패를 예외(`raise ...`) 또는 `sys.exit(비영)`으로 알린다 → **비영(non-zero) 종료코드**. 백엔드는 종료코드가 0이 아니면 실패로 간주한다.

## 2. 호출 표준 코드 형태 (참고 스켈레톤)

```python
import os, subprocess, sys
env = {**os.environ, "PYTHONUTF8": "1", "PYTHONIOENCODING": "utf-8"}
proc = subprocess.run(
    [sys.executable, str(script_path), "--workspace", str(ws), *extra_args],
    capture_output=True, text=True, encoding="utf-8", errors="replace",
    env=env, cwd=str(REPO),   # cwd는 REPO로 두면 스킬 내부 상대참조가 안전
)
# proc.returncode == 0 → 성공, stdout에서 결과 경로 파싱
# != 0 → 실패, stderr를 ingestion_logs에 저장
```

  - `<WS>/비행자료/비행자료검색/비행자료검색_YYYYMMDD.xlsx` (정규식 `비행자료검색_(\d{8})\.xlsx`)
  - `<WS>/공간데이터/FIR.csv` (필수)
  - `<WS>/공간데이터/2026년 공항목록.xlsx` 우선, 없으면 `*공항목록*.xlsx` 중 최신
  - `~$`로 시작하는 Excel 임시파일은 무시
  - **분석·검색 파일은 날짜별 1:1 쌍**이어야 하며 불일치 시 중단 → workspace_builder가 쌍을 보장해야 함

- **stdout 결과 파싱**: `OUTPUT=<경로>` 라인을 파싱(JSON 아님). 그 외 `DATE_RANGE=`, `FILE_PAIRS=`, `MAPPING_RATE=`, `UNMATCHED_IDS=`, 각종 `KEY=VALUE` 통계 라인. 고고도(>60000ft) 경고는 stderr `WARNING=`으로만 나오며 실패 아님.
- **검증기**: `<REPO>/skills/preprocess-flight-data/scripts/validate_output.py`
  - **위치 인자** `csv_path`(필수) + 선택 `--expected-rows <int>`. (플래그가 아니라 위치 인자임에 주의)
  - 메인이 자동 호출하지 않음 → **백엔드가 별도로 실행**
  - 결과: stdout JSON(`path`, `rows`, `leading_zero_ssr`, `utf8_bom` 등). 실패 시 `raise ValueError`(비영 종료)

### 3.2 preprocess-acdm (ACDM)

- **run_type**: `acdm`
- **스크립트**: `<REPO>/skills/preprocess-acdm/scripts/run_acdm_preprocessing.py`
- **CLI 인자**

  | 인자 | 필수 | 기본값 | 백엔드 전달값 |
  |---|---|---|---|
  | `--workspace` | **필수** | 없음 | `<WS>` |
  | `--input-dir` | 선택 | `<WS>/ACDM` | 기본값 사용 |
  | `--output-dir` | 선택 | `<WS>/outputs/acdm_preprocessed` | 기본값 사용 |
  | `--keep-staging` | 선택(flag) | False | 디버깅 시에만 지정 |

- **입력 폴더 구조**: `<WS>/ACDM/` 하위를 재귀 탐색(`rglob`)해 `.csv`/`.xlsx` 전부 수집. **공항·방향은 파일의 상위 폴더명에서 파싱**:
  - 공항: 폴더명이 `인천`/`김포`/`제주`/`김해`로 **시작**하면 → `RKSI`/`RKSS`/`RKPC`/`RKPK`
  - 방향: 폴더명에 `출발` 포함 → 출발, `도착` 포함 → 도착. 둘 다 없으면 ValueError
  - 예: `ACDM/인천공항출발ACDM/...`, `ACDM/김포공항도착ACDM/...` (정확한 이름이 아니라 접두사/부분일치)
  - 파일명 끝 8자리 날짜 필수(정규식 `(\d{8})$` on stem). 없으면 ValueError
  - 인천은 `.csv`(header=1), 지방공항은 `.xlsx`(header=0) 스키마로 구분 처리
  - → **workspace_builder는 업로드 폼에서 받은 공항/방향으로 `<WS>/ACDM/<공항><방향>ACDM/` 폴더명을 재구성**하고 그 안에 날짜 접미 파일을 배치해야 함
- **출력**: `<WS>/outputs/acdm_preprocessed/`
  - `ACDM_출발_전처리_{YYYYMMDD}-{YYYYMMDD}.csv`
  - `ACDM_도착_전처리_{YYYYMMDD}-{YYYYMMDD}.csv`
  - 기간은 `operation_date` 열 min/max. UTF-8 BOM.
  - 최종 저장 전 `timeline_status == "검토필요"` 행은 제외됨
- **stdout 결과 파싱**: 종합 **JSON** 출력. 키: `departure_output`, `arrival_output`(두 최종 경로), `departure input/output rows`, 제외행수, 그리고 검증기 stdout을 파싱한 `validation` 객체 포함.
- **검증기**: `<REPO>/skills/preprocess-acdm/scripts/validate_outputs.py`
  - 인자: `--departure <path> --arrival <path>` (둘 다 필수)
  - **메인이 최종본 생성 후 이 검증기를 subprocess로 자동 실행**한다. 실패면 메인이 RuntimeError로 전체 중단 → 별도 실행 불필요(단, 백엔드가 결과를 다시 확인하고 싶으면 재실행 가능)
  - 검증 내용: BOM, 열 구조가 DEPARTURE(31열)/ARRIVAL(25열)와 정확히 일치, source/iata 잔존 금지, `검토필요` 잔존 금지, FIR→APP 0~90 범위
- **ACDM 내부 동작(참고)**: 메인이 7개 스테이지를 순차 subprocess로 실행하며, `<WS>/outputs/.acdm_preprocess_stage_<uuid10>` 임시폴더를 만들어 격리한다. 성공+`--keep-staging` 미지정 시 임시폴더 삭제, 실패 시 보존하고 stderr에 `STAGING_RETAINED=<경로>` 출력. → **백엔드의 workspace 격리와 무관하게 스킬 자체가 한 번 더 격리**하므로 동시 실행 충돌 위험이 낮다.

- **입력 폴더 구조**:
  - `<WS>/흐름관리일지/*.xlsx` (각 XLSX의 모든 시트에서 상위 20행 내 `Seq/Date/Start/End`가 모두 있는 행을 헤더로 자동 탐지)
  - 필수 열 19개: `Seq, Date, RTime, Sender, Receiver, Reasons, Start, End, Run, MINIT, MIT, Level Capping & G/S, Destination, Route, Fix, Remarks, Dir, CTOT/Note, CTOT Pub`. 누락 시 ValueError
- **출력**: `<WS>/outputs/flow_management_preprocessed/`
  - 항상: `흐름관리일지_전처리_{YYYYMMDD}-{YYYYMMDD}.csv` (UTF-8 BOM)
  - 항상: `흐름관리전처리_검증_{YYYYMMDD}-{YYYYMMDD}.json` (BOM 없는 UTF-8)
  - `--no-integrate` 미지정 시에만 추가(1단계는 해당 없음): `흐름관리영향상세_{기간}.csv`, `비행경로추천통합데이터_흐름관리_{기간}.csv`
  - 기간은 `기록일자`(원본 Date) min/max
  - 흐름관리ID 형식: `FLOW_{YYYYMMDD}_{Seq:04d}` (중복 시 `_2` 접미)
  - **주의**: 이 스킬만 CSV에 `lineterminator`를 지정하지 않아 플랫폼 기본값을 쓴다(다른 3종은 `\n` 고정). 동일성 검증 시 개행문자 차이를 값 비교에서 무시하도록 주의.
- **stdout 결과 파싱**: 종합 **JSON** 출력. 키: `event_output`, `detail_output`, `integrated_output`, `validation_output`, `status: "PASS"`, 각종 통계. (별도 `KEY=VALUE` 라인 없음). 1단계(`--no-integrate`)에서는 `detail_output`/`integrated_output`이 비어 있음.
- **검증기**: `<REPO>/skills/preprocess-flow-management/scripts/validate_outputs.py`
  - 인자: **`--report <검증JSON경로>`** (CSV가 아니라 메인이 만든 검증 JSON을 받음)
  - 메인이 자동 호출하지 않음 → **백엔드가 별도 실행**. 넘길 경로는 stdout JSON의 `validation_output`
  - 결과: stdout JSON `{"status": "PASS"|"FAIL", "failures": [...]}`. FAIL 시 `raise SystemExit(1)` → **종료코드 1**

## 4. 공간데이터(`공간데이터/`) 취급

- flight-data 스킬만 공간데이터를 필요로 한다(출발 FIR 보정). `--spatial-root` 기본값은 `<WS>/공간데이터`.
- 1단계 업로드 UI는 비행자료/ACDM/FOIS/흐름관리 4종만 다루고 공간데이터는 업로드 대상이 아니다. 따라서 **workspace_builder는 flight_data run을 만들 때 `<REPO>/공간데이터/`의 내용을 `<WS>/공간데이터/`로 복사(또는 심볼릭 링크)해 스킬이 찾을 수 있게 해야 한다.**
- 공간데이터는 자주 바뀌지 않으므로, 원본을 수정하지 않는 선에서 read-only 참조로 두는 것을 권장한다.

## 5. workspace_builder 배치 규칙 요약 (run_type별)

| run_type | `<WS>` 안에 만들 구조 | 업로드에서 받아야 할 메타 |
|---|---|---|
| `flight_data` | `비행자료/비행자료분석/비행자료분석_YYYYMMDD.xlsx`, `비행자료/비행자료검색/비행자료검색_YYYYMMDD.xlsx`, `공간데이터/*`(REPO에서 복사) | 날짜별 분석/검색 **쌍** |
| `acdm` | `ACDM/<공항><방향>ACDM/<원본명 또는 재명명>_YYYYMMDD.<csv\|xlsx>` | 공항(인천/김포/제주/김해), 방향(출발/도착), 파일명 끝 8자리 날짜 |
| `fois` | `FOIS/비정상운항출발/*.xlsx` 또는 `FOIS/비정상운항도착/*.xlsx` | 방향(출발/도착) |
| `flow_management` | `흐름관리일지/*.xlsx` | 없음(파일만) |

> ACDM은 파일 stem 끝에 8자리 날짜가 반드시 있어야 하므로, 업로드 파일명에 날짜가 없으면 workspace_builder가 폼에서 받은 날짜로 파일명을 재구성해 배치한다.

## 6. 결과 수집 로직 요약 (run_type별)

| run_type | 최종 CSV 경로 파싱 | 검증기 실행 주체 | 검증기 인자 |
|---|---|---|---|
| `flight_data` | stdout `OUTPUT=` 라인 | 백엔드가 별도 실행 | 위치인자 `csv_path` (+`--expected-rows`) |
| `acdm` | stdout JSON `departure_output`/`arrival_output` | **메인이 자동 실행** | (자동) `--departure --arrival` |
| `fois` | stdout `DEPARTURE_OUTPUT=`/`ARRIVAL_OUTPUT=` 또는 JSON | 백엔드가 별도 실행 | `--departure --arrival` |
| `flow_management` | stdout JSON `event_output` (+`validation_output`) | 백엔드가 별도 실행 | `--report <validation_output>` |

이 표대로 `skill_runner.py`가 run_type을 분기하면 4종을 모두 처리할 수 있다.
