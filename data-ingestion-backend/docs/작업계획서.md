- 문서 버전: 2.0
- 작성일: 2026-07-21

- 관련 문서: [스킬연동_레퍼런스](./스킬연동_레퍼런스.md), [기술스택_결정](./기술스택_결정.md), [DB스키마](./DB스키마.md), [체크리스트](./체크리스트.md)

> 이 문서 묶음은 "코드 없이 문서만 읽고 바로 구현 착수"를 목표로 작성되었다. 스킬의 정확한 CLI/출력/폴더/검증 인터페이스는 [스킬연동_레퍼런스](./스킬연동_레퍼런스.md)에 코드 검증 기반으로 정리되어 있으므로, 구현 전 그 문서를 반드시 먼저 읽는다.

## 1. 배경과 목표

`데이터전처리기술이식` 프로젝트는 항공데이터(비행자료/ACDM/FOIS/흐름관리일지) 전처리 스킬 4개가 이미 구현되어 CSV 산출물을 만들고 있다(`skills/*`, `outputs/*`). 최종 목표는 전처리 결과를 PostgreSQL DB에 적재해 이후 분석/서비스(비행경로 추천 등)의 기반 데이터로 쓰는 것이다.

**1단계 목표**: "파일 수신 → 기존 전처리 스킬 재사용 → DB 적재"를 수행하는 백엔드를 만든다. 백엔드 중심이며, 프론트는 업로드 폼 페이지와 처리결과/로그 조회 페이지 두 개만 만든다. 본격적인 분석/서비스 화면, 통합 스킬(비행자료+ACDM+FOIS 결합) 구현, API 자동 수신은 이번 범위가 아니다.

## 2. 확정 요구사항

1. 파일 수신은 웹페이지에서의 **수동 업로드** (API 자동 수신은 향후 확장 대상).
2. **업로드 즉시 트리거되는 배치 처리** (스트리밍/폴링 아님).
3. PostgreSQL에는 **전처리 완료 데이터 + 원본(raw) 데이터를 모두 별도 테이블**에 적재.
4. DB는 **로컬 Docker PostgreSQL로 시작**하고, **향후 클라우드 Supabase로 이전**하는 것을 전제로 설계.
5. UI는 백엔드 중심 — 업로드 폼 + 로그 조회 페이지만.
6. 파싱/전처리 로직은 새로 만들지 않고 **이 프로젝트 폴더의 기존 `skills/*/scripts`를 그대로 재사용**한다(원본 데이터는 각 원본 폴더에서 읽는다).
7. 업로드 입력값 검증(확장자/용량/필수 쌍파일)을 서버에서 강제한다.
8. 동일 요청 재시도를 위한 멱등키(`idempotency_key`)를 지원한다.
9. **표준화·저(低)하드코딩 원칙**: 스킬마다 다른 인터페이스(§4)는 선언적 서술자로 데이터화하고, 실행/적재 코드는 그 서술자를 따라 동작하는 **하나의 일반화된 경로**로 구현한다. run_type별 특수 분기와 매직 문자열(경로·컬럼명·정규식)은 서술자/설정/상수 모듈에 모으고 로직 본문에서 최소화한다.

## 3. 전체 아키텍처

```text
[사용자] → 업로드 폼(Jinja2) → POST /uploads
                                      │
                                      ▼
                    ┌─────────────────────────────────┐
                    │  1. 입력검증 + raw_files/raw_*   │  (원본 그대로 보존)
                    └─────────────────────────────────┘
                                      │
                                      ▼
                    ┌─────────────────────────────────┐
                    │  2. workspace_builder            │  SkillDescriptor.layout에
                    │     (run별 격리 임시 폴더 구성)   │  따라 파일 배치
                    └─────────────────────────────────┘
                                      │  BackgroundTasks.add_task
                                      ▼
                    ┌─────────────────────────────────┐
                    │  3. skill_runner (subprocess)    │  SkillDescriptor.command/
                    │     + result_parser + validator  │  parse/validate 서술자 사용
                    └─────────────────────────────────┘
                                      │  성공 + 검증통과
                                      ▼
                    ┌─────────────────────────────────┐
                    │  4. loaders (processed_* 적재)   │  SkillDescriptor.tables 사용
                    └─────────────────────────────────┘
                                      │
                                      ▼
                    ingestion_runs 상태 갱신 → 로그 페이지(GET /runs, /runs/{id})
```

## 4. 표준화 설계 — SkillDescriptor 레지스트리 (핵심)

스킬 4종은 CLI 인자, 입력 폴더 규칙, stdout 결과 파싱 방식, 검증기 호출 방식, 날짜 구분자, 산출물→DB 테이블 매핑이 모두 다르다([스킬연동_레퍼런스](./스킬연동_레퍼런스.md)). 이 차이를 **코드 분기 대신 하나의 선언적 서술자 객체**로 표현한다. 서술자는 `app/ingestion/registry.py`에 상수로 모아 두고, 실행·적재 코드는 서술자를 인자로 받아 동일한 절차를 반복한다.

### 4.1 SkillDescriptor 구조 (dataclass, frozen)

```python
@dataclass(frozen=True)
class UploadSlot:
    key: str               # 폼 필드/논리 슬롯 이름 (예: "analysis", "search", "file")
    required: bool
    accept: tuple[str,...] # (".xlsx",) 또는 (".csv",".xlsx")
    paired: bool = False   # 날짜별 쌍 요건(비행자료 분석/검색)

@dataclass(frozen=True)
class LayoutRule:
    # workspace_builder가 <WS> 안에 파일을 놓을 위치를 만드는 규칙.
    # 템플릿에 {airport},{direction},{date},{filename} 치환자 사용.
    subdir_template: str          # 예: "ACDM/{airport_ko}{direction_ko}ACDM"
    filename_template: str | None # None이면 원본 파일명 유지, 아니면 재명명
    needs_spatial: bool = False   # 공간데이터 복사 필요 여부(flight_data=True)

@dataclass(frozen=True)
class ResultParse:
    mode: str                     # "line" | "json"
    keys: tuple[str,...]          # line모드: ("OUTPUT",) / json모드: ("departure_output","arrival_output")

@dataclass(frozen=True)
class ValidatorSpec:
    auto_run_by_skill: bool       # acdm=True(메인이 자동 실행), 나머지=False
    script: str | None            # 별도 실행 시 검증기 스크립트 상대경로
    arg_style: str | None         # "positional_csv" | "dep_arr" | "report"

@dataclass(frozen=True)
class OutputTableMap:
    # 산출물 논리키 → processed 테이블명 + 컬럼매핑 이름
    logical_key: str              # ResultParse.keys 중 하나
    table: str                    # 예: "processed_acdm_departure"
    column_map: str               # DB스키마.md의 매핑 테이블 식별자

@dataclass(frozen=True)
class SkillDescriptor:
    run_type: str                 # "flight_data"|"acdm"|"fois"|"flow_management"
    script: str                   # <REPO> 기준 상대경로
    extra_args: tuple[str,...]    # 예: flow_management=("--no-integrate",)
    upload_slots: tuple[UploadSlot,...]
    layout: LayoutRule
    date_separator: str           # flight_data="_" / 나머지="-"
    result: ResultParse
    validator: ValidatorSpec
    outputs: tuple[OutputTableMap,...]
    raw_tables: tuple[str,...]    # 이 run_type이 적재하는 raw_*_rows 테이블
```

### 4.2 4종 서술자 값 (요약, 근거는 스킬연동_레퍼런스 §3)

| 필드 | flight_data | acdm | fois | flow_management |
|---|---|---|---|---|
| `script` | `skills/preprocess-flight-data/scripts/preprocess_flight_data.py` | `skills/preprocess-acdm/scripts/run_acdm_preprocessing.py` | `skills/preprocess-fois/scripts/run_fois_preprocessing.py` | `skills/preprocess-flow-management/scripts/run_flow_management_preprocessing.py` |
| `extra_args` | `()` | `()` | `()` | `("--no-integrate",)` |
| `date_separator` | `_` | `-` | `-` | `-` |
| `result.mode` | `line` | `json` | `line` | `json` |
| `result.keys` | `("OUTPUT",)` | `("departure_output","arrival_output")` | `("DEPARTURE_OUTPUT","ARRIVAL_OUTPUT")` | `("event_output",)` |
| `validator.auto_run_by_skill` | `False` | `True` | `False` | `False` |
| `validator.arg_style` | `positional_csv` | `dep_arr` | `dep_arr` | `report` |
| `layout.needs_spatial` | `True` | `False` | `False` | `False` |
| `outputs` | `processed_flight_data` 1종 | 출발/도착 2종 | 출발/도착 2종 | `processed_flow_management` 1종 |

> 이렇게 하면 `skill_runner`·`loaders`·`workspace_builder`는 "서술자를 받아 처리"만 하면 되고, 새 스킬이 추가되어도 서술자 한 항목만 등록하면 된다. 하드코딩되는 매직값은 이 레지스트리와 `constants.py`(폴더명·정규식·공항코드 매핑 등)에만 존재한다.

### 4.3 하드코딩 최소화 세부 규칙

- 원본 폴더명(`비행자료`, `ACDM/…`, `FOIS/비정상운항출발` 등), 공항 한글↔ICAO 매핑, 파일명 날짜 정규식 등 "스킬이 강제하는 계약값"은 **스킬 코드가 원본(진실의 소스)**이므로 백엔드는 이를 복제하되 `constants.py` 한 곳에 모으고, 값이 스킬과 어긋나면 통합테스트(§10)에서 즉시 드러나게 한다.
- DB 컬럼 물리명은 [DB스키마](./DB스키마.md)의 매핑 테이블을 **단일 출처**로 삼아 `models.py`와 `loaders.py`가 같은 매핑을 공유한다(양쪽에 문자열을 중복 정의하지 않는다).
- 환경값(경로 루트, 최대 업로드 크기, 허용 확장자, DB URL)은 `config.py`의 `Settings`로만 읽고 코드에 리터럴로 박지 않는다.

## 5. 처리 파이프라인 (서술자 기반, 9단계)

1. **업로드 접수/검증**: 폼에서 (a) 처리유형, (b) ACDM이면 공항+방향(+날짜), (c) 파일 첨부 → `POST /uploads`. 서버는 `SkillDescriptor.upload_slots`로 확장자·필수·쌍요건을, `Settings`로 용량을 검증한다. 위반 시 run 생성 전에 `400`.
2. **원본 보존 + raw 적재**: sha256 계산 후 `data/uploads/<run_id>/`에 원본 저장 → `raw_files` insert. pandas로 원본을 읽어 `SkillDescriptor.raw_tables`의 `raw_*_rows`에 계약 컬럼 + `extra_columns`로 insert.
3. **workspace 준비**: `ingestion_runs`(`status=QUEUED`) 생성 → `<WS>=data/workspace/<run_id>/` 생성 → `SkillDescriptor.layout`에 따라 파일 배치(ACDM 폴더명 재구성, 비행자료 쌍 배치, `needs_spatial`이면 `공간데이터/` 복사).
4. **트리거/멱등**: `BackgroundTasks.add_task(run_ingestion, run_id)` 후 즉시 응답. `idempotency_key`가 있고 동일 키 run이 `QUEUED/RUNNING/SUCCESS`면 새 run 생성 없이 기존 run 반환. 이미 `RUNNING`인 run이 있으면 이번 run은 `QUEUED`로 두고 순차 처리.
5. **스킬 실행**: `status=RUNNING` → [스킬연동_레퍼런스](./스킬연동_레퍼런스.md) §2의 표준 형태로 subprocess 실행(`--workspace <WS>` + `extra_args`, `PYTHONUTF8/IOENCODING` 주입). stdout/stderr를 `ingestion_logs`에 저장.
6. **결과 파싱 + 검증**: `SkillDescriptor.result`로 최종 CSV 경로 파싱. `validator.auto_run_by_skill`이 False면 `validator.arg_style`대로 검증기를 별도 실행. 검증 통계를 `validation_summary`에 저장. 검증 실패 시 `status=VALIDATION_FAILED`, processed 적재 생략(원본/중간산출물 보존).
7. **Processed 적재**: 검증 통과 시 `SkillDescriptor.outputs`대로 각 CSV를 pandas로 읽어 해당 `processed_*` 테이블에 `run_id`와 함께 적재. raw 적재와는 **별도 트랜잭션**(§9-4).
8. **완료**: `status=SUCCESS`, `finished_at`, `duration_ms` 기록. `<WS>`의 중간 스테이징 정리(원본 아카이브·최종 CSV는 보존).
9. **로그 조회**: `GET /runs`(목록·상태뱃지), `GET /runs/{id}`(입력파일·검증요약·실패로그·CSV 다운로드).

## 6. 모듈별 구현 명세

> 폴더 구조는 §8. 아래는 각 모듈의 책임과 대표 시그니처(구현 착수용 청사진).

- **`app/config.py`** — `Settings`(dataclass): `database_url`, `upload_dir`, `workspace_root`, `repo_root`, `max_upload_mb`, `allowed_extensions`. `python-dotenv`로 로드.
- **`app/db/session.py`** — SQLAlchemy 2.0 Engine/Connection 팩토리. DB 접근을 이 파일로 캡슐화(드라이버 교체 지점).
- **`app/db/models.py`** — SQLAlchemy Core `Table` 정의(모든 테이블). 물리 컬럼명은 [DB스키마](./DB스키마.md) 매핑을 그대로 따른다.
- **`app/ingestion/registry.py`** — §4의 `SkillDescriptor` 4종 상수. `get_descriptor(run_type) -> SkillDescriptor`.
- **`app/ingestion/constants.py`** — 원본 폴더명, 공항 한글↔ICAO(`{"인천":"RKSI",...}`), 방향 토큰, 파일명 날짜 정규식 등 스킬 계약 매직값.
- **`app/ingestion/workspace_builder.py`** — `build_workspace(run, descriptor, uploaded_files) -> Path`. 서술자 `layout`에 따라 `<WS>` 구성. 반환: workspace 경로.
- **`app/ingestion/skill_runner.py`** — `run_skill(descriptor, ws) -> SkillResult`(returncode/stdout/stderr/parsed_paths). `parse_result(descriptor, stdout)`, `run_validator(descriptor, paths) -> ValidationSummary`.
- **`app/ingestion/loaders.py`** — `load_raw(run, descriptor, uploaded_files)`, `load_processed(run, descriptor, output_paths)`. 컬럼 매핑은 [DB스키마](./DB스키마.md)의 매핑을 공유.
- **`app/ingestion/pipeline.py`** — `run_ingestion(run_id)`. §5의 5~8단계를 서술자로 오케스트레이션. 상태 전이·예외→`error_code` 매핑.
- **`app/routers/upload.py`** — `POST /uploads`, `GET /`(업로드 폼). 입력검증·raw적재·workspace준비·트리거.
- **`app/routers/runs.py`** — `GET /runs`, `GET /runs/{id}`, `GET /runs/{id}/download/{artifact}`.
- **`app/main.py`** — FastAPI 앱, 라우터 등록, `Jinja2Templates`, 정적파일.

## 7. 업로드 보안/유효성 최소 기준

1. 허용 확장자만 수락: `.csv`, `.xlsx` (`Settings.allowed_extensions`).
2. 업로드 파일당 최대 크기 제한(기본 200MB, `Settings.max_upload_mb`).
3. 저장 파일명은 서버 생성명(uuid) 사용, 사용자 파일명은 `raw_files.original_filename`에만 보관(경로주입 방지).
4. 경로 문자열(`../`) 포함 입력은 즉시 거부.
5. 비행자료는 날짜별 분석/검색 쌍 검증 실패 시 run 생성 전 `400 Bad Request`.

## 9. 리스크 / 열린 이슈

1. **대용량 업로드 처리시간**: flight-data는 shapely 공간연산(FIR 판정)을 포함해 상대적으로 무겁다. 단일 실행 가드만으로는 업로드가 몰릴 때 대기시간이 길어진다 → 로그 페이지에 `QUEUED/RUNNING` 상태와 시작시각을 노출. 빈도 증가 시 Celery/RQ 전환 검토.
2. **ACDM 공항/방향 판별**: 스킬이 폴더명 접두사(`인천/김포/제주/김해`)와 부분일치(`출발/도착`)로 판별하므로, 업로드 폼에서 공항/방향/날짜를 반드시 받아 `workspace_builder`가 폴더명·파일명을 재구성해야 한다.
3. **동시성/workspace 충돌**: run마다 완전히 격리된 `<WS>`를 쓴다(§5-3). ACDM은 스킬 자체도 uuid 스테이지 폴더로 한 번 더 격리한다.
4. **트랜잭션 분리**: raw 적재와 processed 적재를 별도 트랜잭션으로 처리해, 후반 실패가 원본 추적성까지 롤백하지 않게 한다.
5. **흐름관리 상류 의존성 미완성**: 통합 스킬(`비행경로추천통합데이터.csv` 생성)이 아직 없어 1단계는 **항상 `--no-integrate`**로 흐름관리 자체 전처리만 지원한다. 영향상세/통합 결합은 다음 단계([DB스키마](./DB스키마.md) 4.1).
6. **검증실패/검토필요 UI 노출**: ACDM `검토필요` 행은 최종 CSV에서 제외되어 DB에 안 들어오지만, 제외 건수를 `validation_summary`에 저장하고 로그 상세에 경고 배지로 노출한다.
7. **Supabase 이전 변경점**: 연결문자열(direct vs pooler), `sslmode=require`, RLS 비활성 유지, 커넥션 풀 제한, 파일 저장소(볼륨 vs Supabase Storage). Alembic 마이그레이션은 direct connection으로 실행. 상세는 [DB스키마](./DB스키마.md) 7장.
8. **중복 업로드 정책**: 동일 sha256 재업로드 시 경고/차단 정책을 확정한다([DB스키마](./DB스키마.md)의 `raw_files` unique 인덱스와 연동).
9. **인증정보 관리**: `.env`(DB 비밀번호 등)는 커밋 금지(`.gitignore`). API 자동수신 확장 대비 인증키 하드코딩 금지.
10. **fois 분류사전/flow 매핑 JSON 존재 확인**: `fois-taxonomy-v0.1.json`, `scope-mapping.json`이 스킬 폴더에 실존해야 한다(없으면 실패). 배포 체크리스트에 포함.
11. **실측 성능 부재**: 현재 샘플은 1개월치 규모. 통합테스트에서 실제 규모에 가까운 데이터로 처리시간을 1회 이상 측정.
12. **개행문자 차이(flow만)**: flow-management 산출 CSV는 `lineterminator` 미지정이라 플랫폼 기본 개행을 쓴다. 동일성 검증(§10)에서 값 비교 시 개행 차이를 무시하도록 한다.

## 10. 완료 기준 (Definition of Done) — 실제 원본데이터 동일성 검증

**개발 완료로 보려면, 이 프로젝트 폴더에 실제로 존재하는 원본 폴더(`비행자료/`, `ACDM/`, `FOIS/`, `흐름관리일지/`)를 백엔드로 처리한 결과가, 동일 입력 기준의 검증된 결과와 완전히 일치함을 확인해야 한다.** "에러 없이 끝났다"만으로는 완료가 아니다.

### 10.1 기준선(ground truth)

| 데이터 | 기준선 |
|---|---|
| 비행자료 | 기존 `outputs/flight_data_preprocessed/` 산출물(원본이 그 이후 안 바뀌었다면 유효) |
| FOIS | 기존 `outputs/fois_preprocessed/` 산출물 |
| 흐름관리일지 | 기존 `outputs/flow_management_preprocessed/` 산출물(`--no-integrate` 기준) |
| **ACDM** | 기존 `outputs/acdm_timeline_fixed/`는 **사용하지 않는다**(`검토필요` 139행 이슈). 최신 `preprocess-acdm`을 원본 `ACDM/`에 **직접 CLI로 재실행**한 결과를 기준선으로 새로 만든다 |

기준선은 모두 "백엔드를 거치지 않고 스킬을 직접 CLI로 실행한 결과"다. 백엔드가 검증할 것은 "백엔드를 거쳐도 결과가 달라지지 않는다"이므로, 기준선이 백엔드 코드에 의존하면 안 된다.

### 10.2 검증 절차

1. 원본 폴더 4종을 백엔드 업로드 화면으로 각각 처리한다.
2. 백엔드 `processed_*` DB 값 ↔ 백엔드가 적재에 쓴 스킬 최종 CSV 비교 → **DB 로더가 값을 훼손하지 않았는지** 확인.
3. 백엔드 처리 결과(최종 CSV 또는 DB) ↔ 10.1 기준선(직접 CLI 실행) 비교 → **웹 업로드→workspace 재구성→subprocess 경로가 직접 실행과 동일 결과를 내는지** 확인.
4. 통과 기준: 행 수 정확히 일치 + 자연키 정렬 후 전체 컬럼 값 일치(정렬키 — 비행자료: `고유ID`, ACDM: `operation_date+airport_icao+flight_icao`, FOIS: `편명+출발/도착일자`, 흐름관리: `흐름관리ID`) + 제외행(검토필요 등) 건수 일치(`validation_summary` 비교). flow-management는 개행문자 차이를 무시.
5. 비교는 `CLAUDE.md` 규칙대로 파일 전체를 직접 열지 않고 짧은 pandas 스크립트(`merge`+`compare` 또는 `assert_frame_equal`)로 수행하고, 스크립트와 결과(일치 여부·불일치 건수)만 기록.
6. 불일치 시 원인(원본 변경 vs 파이프라인 결함)을 특정하기 전까지 완료로 보지 않는다.

이 검증은 [체크리스트](./체크리스트.md) Phase 8에 실행 단위로 반영되어 있다.
