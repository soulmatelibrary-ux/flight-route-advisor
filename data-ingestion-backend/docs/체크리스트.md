- [ ] `workspace_builder.py`: `build_workspace(run, descriptor, files)` — 서술자 `layout`에 따라 `<WS>` 구성(ACDM 폴더명/파일명 재구성, 비행자료 쌍 배치, `needs_spatial`이면 공간데이터 복사)
- [ ] `skill_runner.py`: `run_skill`(subprocess, `PYTHONUTF8/IOENCODING` 주입, `--workspace`+`extra_args`), `parse_result`(서술자 `result` 기반 line/json 파싱), `run_validator`(서술자 `validator.arg_style`별 분기)

- [ ] flow-management 호출에 `--no-integrate`가 항상 포함되는지 확인

### 8.1 실제 원본데이터 동일성 검증 ([작업계획서.md](./작업계획서.md) 8장 참고, 개발완료 판정에 필수)

- [ ] ACDM 기준선 생성: 백엔드와 무관하게 최신 `preprocess-acdm` 스킬을 원본 `ACDM/` 폴더에 **직접 CLI로 재실행**해 기준선 CSV를 새로 만든다 (기존 `outputs/acdm_timeline_fixed/`는 검토필요 139행 이슈가 있어 기준선으로 쓰지 않음)
- [ ] 비행자료/FOIS/흐름관리일지 기준선 확인: 기존 `outputs/flight_data_preprocessed/`, `outputs/fois_preprocessed/`, `outputs/flow_management_preprocessed/`가 현재 원본 폴더 상태와 여전히 일치하는지 확인(원본이 그 사이 바뀌지 않았다면 그대로 기준선으로 사용)
- [ ] 원본 폴더 4종(`비행자료/`, `ACDM/`, `FOIS/`, `흐름관리일지/`)을 백엔드 업로드 화면으로 각각 처리
- [ ] 백엔드가 만든 `processed_*` DB 테이블 값과, 백엔드가 적재에 사용한 스킬의 최종 CSV를 pandas로 비교해 **DB 로더가 값을 훼손하지 않았는지** 확인(행수 일치 + 자연키 기준 정렬 후 전체 컬럼 값 일치)
- [ ] 백엔드 처리 결과(2번 CSV 또는 DB)와 위에서 만든 기준선을 pandas로 비교해 **웹 업로드→workspace 재구성→subprocess 실행 경로가 직접 CLI 실행과 동일한 결과를 내는지** 확인(행수 일치 + 자연키 기준 정렬 후 전체 컬럼 값 일치)
- [ ] 검증에서 제외된 행(ACDM `검토필요` 등)의 건수가 백엔드 결과와 기준선에서 동일한지 `validation_summary` 기준으로 확인
- [ ] 비교 스크립트는 CSV/DB 전체를 직접 열어보지 않고 짧은 pandas 스크립트(`merge`+`compare` 또는 `assert_frame_equal`)로 작성하고, 실행 결과(일치 여부, 불일치 건수)를 기록
- [ ] 불일치 발견 시 원인(원본 변경 vs 파이프라인 결함)을 특정하기 전까지 이 항목을 완료로 체크하지 않음
