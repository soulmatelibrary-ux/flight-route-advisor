## 0. 이 문서의 목적과 다음 단계

**사용자 목표(원문 요약)**: 최종 비행경로를 추천할 때 **FOIS·흐름관리·기존 경로(ODR2)를 통합
확인**해 가장 효과적인 노선을 사용자에게 알려주고, **그 이유를 AI로 간략히 생성**해서
**경로를 선택하면 팝업으로 보여준다.** 단, **현재 AI API가 없으므로 1단계 범위는**:
경로 자료 등으로 **프롬프트를 만들어 AI에 입력 → 답변을 받아 결과를 보는 것까지**만.

이 문서는 (1) 지금 무엇이 이미 있는지, (2) 목표까지 무엇이 더 필요한지, (3) 기존 구현 중
무엇을 고쳐야 하는지, (4) 어떤 방향 선택지가 있는지를 정리한다. **결정이 필요한 항목은
§12에 모아** 사용자가 개선요구서에서 답하도록 남긴다.

> 진행 절차(사용자 확정 흐름): **이 분석 MD 검토 → 개선요구서 작성 → (그때) docs 정식 반영 +
> [07-checklist](./07-checklist.md) 항목화 → 구현 → 리뷰에이전트 게이트**([CLAUDE.md](../CLAUDE.md) §4).

## 1. 1단계 범위 재정의 (이번에 "완료"로 볼 것)

AI API가 아직 없다는 제약을 그대로 반영하면, **이번 1단계의 "done" 정의**는:

1. 노선(OD) + 후보 경로를 선택하면, 그 경로에 대한 **FOIS·흐름관리·경로지표를 하나의 맥락으로
   통합**한다.
2. 그 통합 맥락으로 **AI에 넣을 프롬프트(한국어, 관제 도메인)를 생성**한다.
3. 그 프롬프트를 **AI에 입력 → 답변을 받아** 오는 경로를 만든다. *(AI API가 없으므로 이 "입력→답변"
   경로를 어떻게 채울지가 핵심 결정 → §7.)*
4. 받은 답변(추천 노선 + 이유)을 **경로 선택 시 팝업으로 표시**한다.

여기에 **가볍게 얹을 수 있는 이미-가용 데이터**(§0.1 표 '1단계' 칸): 출/도착 공항 기상 바람·강수(§15),
공항 운항 KPI(§16 기구현 `/ops`), 흐름관리 "통과시점 발효중" 유효창 필터(§14). 반대로 **교통량 프로파일
(§13)·항로 상공 상층풍/시어(§15 ①)·활주로별 혼잡(§16)·실시간 메시지 수신(§14)·완전 세그먼트 병목
(§17)** 은 후속이다.

> **명시적 비범위(이번 1단계 아님)**: 실제 상용 AI API 상시 연동, AI가 경로 순위를 자동
> 재정렬해 확정하는 로직, FOIS 원인을 FIR/항로 구간에 인과적으로 매핑하는 것
> ([07-checklist](./07-checklist.md) "AI 경로추천 근거화" 계획의 **후속 범위** — 1단계 아님).

## 2. 현재 구현 현황 — 재사용 가능한 자산

> ⚠ **현황 스냅샷 주의(2026-07-23)**: 동시 세션이 활발해 아래 "현황"은 빠르게 변한다. 특히 사용자
> 2번째 스크린샷의 **풍부한 경로별 표시(정시성·지연%·흐름관리%·실적고도·상층풍·난류·강수레이더·
> ops KPI)는 "완성본 원본(정답지)"의 것**이고, advisor는 아직 포팅 중이다(ops 어댑터 `onTimeRate`·
> `layers/radar.js`·`layers/hazards.js`·`ops-panel.js` 동시세션 진행, 경로별 정시성/흐름관리%/상층풍
> 브리핑은 advisor route-panel에 **미포팅**). 개선요구서 확정 전 실제 파일로 재확인할 것.

목표에 필요한 **데이터 소스 3종이 이미 다 구현**돼 있다. 문제는 이들이 **경로(OD·옵션) 단위로
합쳐져 있지 않다**는 것.

### 2.1 경로추천 (ODR2) — 있음
- 백엔드 `GET /api/routes?dep&arr` → `backend/app/queries/routes.py:routes_for()`.
- 옵션 1건이 담는 지표(`_shape_option`): `flights`(편수), `avg_min`(평균소요), `delay_count`(지연수),
  `heavy_count`(HEAVY수), `enroute_firs`(경유 FIR 배열), `incheon_track_fixes`(인천 픽스체인),
  `track_coords`/`full_route_coords`(좌표), `cruise_parity`.
- 프론트 `frontend/js/route-panel.js`가 이미 **간이 추천 표기**를 한다:
  - "**권장**" = 운항편수 점유율 최대 옵션(`renderBriefing`).
  - "**[최단]**" = `avgMin` 최소 옵션(`renderOptions`).
  - 즉 **결정 로직의 씨앗은 이미 있으나 단일지표 기반**이고 FOIS/흐름관리는 반영 안 됨.
- 상태 관리 `frontend/js/store.js`: `selectOd()`가 `state.routeResult`(옵션 배열)를 채우고,
  `selectOption(index)`는 **지도 강조 + 목록 재렌더만** 한다(팝업 없음 → §3-4).

### 2.2 FOIS 지연원인 — 있음 (공항 단위)
- 백엔드 `GET /api/fois/delays?direction&airport&date_from&date_to` → `queries/fois.py:delays()`.
- 반환: `total`, `causes[]{cause_major,cause_minor,cause_process,involved_party,reason,count}`.
- 프론트 자산 2개:
  - `frontend/js/fois-panel.js`: 사용자가 **수동 조회**하는 범용 패널(공항·기간 임의).
  - `frontend/js/route-fois-summary.js`: **노선 선택 시 자동으로** 출/도착 공항의 상위 3개 원인을
    "의사결정 근거" 패널(`#route-fois-summary`)에 표시. **← 통합 프롬프트에 가장 가까운 기존 코드.**
- ⚠ **구조적 한계(중요)**: FOIS는 **"공항" 단위 집계이지 "노선(OD)" 단위가 아니다.** 그래서 지금도
  "참고 정보"로만 노출하고 "그래서 이 경로를 추천한다"는 **인과관계로는 쓰지 않는다**
  ([route-fois-summary.js](../frontend/js/route-fois-summary.js) 헤더 주석,
  [07-checklist](./07-checklist.md) 향후확장 계획).

### 2.3 흐름관리 조치 — 있음 (FIR/항로/픽스 단위, 수동)
- 백엔드 `GET /api/flow-management?date_from&date_to&fir&airway&limit&offset` →
  `queries/flow_management.py:list_flow_management()`.
- 반환 항목: `applyStartDt/EndDt`, `applyMinutes`, `minit`, `mit`, `altSpeedLimit`, `targetAirport`,
  `targetFir`, `targetRoute`, `targetFix`, `restrictionSummary`, `qualityStatus`.
- 필터 `fir`/`airway`는 **완전일치 단일값**(`target_fir`/`target_route`).
- 프론트 `frontend/js/flow-management-panel.js`: 역시 **수동 조회** 독립 패널. **노선 선택과 연동 안 됨.**

### 2.4 의사결정 근거 패널 · 팝업 인프라 — 있음
- 우측 패널 `#panel-right`(index.html)에 "의사결정 근거"(`#fir-chain` + `#route-fois-summary`),
  "FOIS 지연원인", "흐름관리 조치" 3블록이 이미 있다.
- 팝업 인프라: `#weather-popup-root`·`#toast-root` div 존재, `frontend/js/weather.js`가 Leaflet 팝업
  패턴(`map.on("popupopen")`)을 씀. `frontend/js/html.js:escapeHtml`로 XSS 방어. **경로 선택 팝업은
  이 패턴을 재사용**할 수 있음.
- 설정: `frontend/js/config.js`·`config.example.json`에 **`externalApis: {}` 플레이스홀더 존재** →
  AI 엔드포인트 설정의 자연스러운 자리(단, 키는 프론트 금지 — §7 보안).

### 데이터 소스별 단위·연결키 (핵심 표)
| 소스 | 집계 단위 | 경로에 붙일 연결키 | 지금 자동연동? |
|---|---|---|---|
| ODR2 경로옵션 | **노선(OD)·옵션** | (기준) | ✅ selectOd |
| FOIS 지연원인 | **공항**(dep/arr) | `dep`/`arr` ICAO | ✅ route-fois-summary(표시만) |
| 흐름관리 조치 | **FIR/항로/픽스** | 옵션의 `enroute_firs[]`·`incheon_track_fixes[]` | ❌ 미연동 |

> ⚠ **연결키 정밀 주의**: 옵션은 `enroute_firs`(FIR)와 `incheon_track_fixes`(픽스)는 주지만 **항로(airway)
> ident는 노출하지 않는다.** 따라서 흐름관리를 경로에 붙일 때 `target_fir`(FIR별)·`target_fix`(픽스별)
> 매칭은 가능하지만 **`target_route`(항로별) 매칭은 현재 데이터로 불가**하다. 이건 §12 결정사항.

## 5. 제안 아키텍처 — 데이터 흐름 & 모듈 배치

```
[옵션 클릭]
  store.selectOption(idx)
     │  (신규) 통합맥락 조립
     ▼
  route-context.js  ── 경로옵션(state.routeResult.options[idx])
     │                ── FOIS dep/arr (state에 저장된 것 재사용 → §4)
     │                ── 흐름관리(경유 FIR별, api.flowManagement per FIR) ── §N5
     ▼
  ai-prompt.js  (프롬프트 빌더, §6)
     │
     ▼
  ai-client.js  (§7의 택1: 수동 복붙 / 목업 / 백엔드 프록시)
     │
     ▼
  route-recommend-popup.js  (§N4: 모달/팝업 렌더 — escapeHtml 필수)
```

- **신규 프론트 모듈(안)**: `route-context.js`(통합기), `ai-prompt.js`(빌더), `ai-client.js`(응답계층),
  `route-recommend-popup.js`(팝업). 기존 `api.js`/`adapters.js`/`config.js` 규약을 그대로 따름.
- **AI를 실제로 서버에서 호출하게 될 때만**(§7-C) 백엔드에 `POST /api/ai/route-reasoning` 추가.
  1단계 §7-A/B는 백엔드 무변경.

## 6. 프롬프트 설계 초안 (§N2)

프롬프트에 넣을 후보 필드(통합기 N1 산출):

- **노선**: dep/arr ICAO, 총 편수.
- **후보 경로들**(옵션별): 경로번호, 편수·점유율, 평균소요(+최단대비 Δ), 지연수, HEAVY수,
  경유 FIR 체인, (있으면) 인천 픽스체인.
- **FOIS 요약**: 출발/도착 공항의 상위 원인 `cause_major`(+세부 `cause_process`) 및 건수.
  ⚠ **자유텍스트 `reason`은 프롬프트에서 제외/마스킹**(민감정보 유입 방지 — 조직정책 §9).
- **흐름관리**: 경유 FIR/픽스에 걸린 조치 요약(`restrictionSummary`, `mit`/`minit`, 적용시간대,
  대상 FIR/항로/픽스). 인과가 아니라 "해당 FIR에 이런 제약이 있었다"는 **맥락**으로 서술.

프롬프트 출력 형태(예시 요청): "가장 효과적인 노선 1개 + 2~3문장 근거(운항빈도/소요/지연/흐름관리
제약을 종합), 자문(advisory)임을 명시." **JSON 스키마 응답을 요구**하면 팝업 렌더가 쉬움
(`{recommended_route, reason, cautions}`).

> 톤 규약: FOIS는 공항 단위·상관관계임을 프롬프트에 명시해 AI가 **인과로 과장하지 않도록** 가드
> (route-fois-summary의 기존 원칙 계승).

## 7. AI 응답 계층 — 3안 비교 (핵심 결정, §N3)

| **A. 수동 복붙**(권장) | 팝업에 완성 프롬프트 표시 + "복사" → 사용자가 외부 AI(예: Google AI Studio)에 붙여넣고 답변을 다시 붙여넣으면 팝업이 렌더 | ★ 사용자 원문("프롬프트 만들어 AI에 입력→답변 받아 결과 보기")과 **정확히 일치**, 즉시 구현 | 키 노출 0, **사람이 검토**(조직정책 부합) | AI API 불요. 빌더+팝업은 C로 갈 때 그대로 재사용 |

> API 연동은 반드시 **백엔드 프록시(C)** 를 거친다([CLAUDE.md](../CLAUDE.md) §3 시큐어코딩, §2 하드코딩 금지).

## 13. (추가 검토) 교통량·혼잡도 반영 — 예측 가능성 분석

> 사용자 질문(2026-07-23): 경로 추천 시 "내 항공기가 지나갈 시점의 항로/주변 교통량"도
> 고려하면 완성도가 오른다. 예측에 필요한 정보가 있는가?

**결론 요약**: **역사적 수요(demand) 프로파일을 만들 재료는 이미 상당히 있다.** 다만
"혼잡도(=수요/용량)"로 정규화하거나 "미래 특정 편의 통과시점"을 실시간 예측하려면 빠진
조각이 있어, 이번 1단계 AI 프롬프트 MVP와는 분리해 후속으로 두길 권장(§12 D10/D11).

### 13.1 이미 가용 (재료 있음)
- **`processed_flight_data`** (현재 advisor API 미노출) — 핵심 소스.
  - `entry_fir`/`entry_datetime`, `exit_fir`/`exit_datetime`: 각 편이 (주로 국내/인천) FIR에 머문
    시간창 → **시간대 버킷 점유수(occupancy)** 집계 가능 = "그 FIR가 몇 시대에 얼마나 붐볐나"의
    역사적 프로파일.
  - `eobt`/`atd`(출발시각 분포), `fir_enroute`/`full_route`(경유 FIR·경로), `rfl/afl/xfl/cfl`(고도).
- **ACDM**(dep/arr): `sobt/tobt/ttot/ctot/aobt`(출발) · `eldt/aldt/sibt/aibt`(도착) →
  **공항 단위 시간대 수요 피크**(도착·출발 러시).
- **흐름관리**(flow): MIT/MINIT/rate 제한 = 과거에 혼잡을 관리한 지점·시간대 = **혼잡의 간접 라벨**.
- **ADS-B**(실시간, 프론트 `adsb.js`): "지금 이 순간" 주변기 — 미래 통과시점 예측은 아니지만
  현재 상황 스냅샷(반경 250nm, 12s 폴).
- **SystemTrack CAT62**(380MB 레이더, 프로젝트 루트 `SystemTrack_CAT62.*.txt`, **미전처리**):
  타임스탬프+위경도+고도+속도 트랙 → 임의 FIR/섹터/항로 폴리곤의 시공간 점유를 세밀히 산출 가능.
  단 **전처리 스킬 필요**(현재 processed_* 미적재, 대용량 — [CLAUDE.md](../CLAUDE.md) §6 직접 열기 금지).

### 13.2 빠진 조각 (예측 정밀도의 한계)
1. **용량(capacity) 기준 부재**: 섹터/FIR 선언용량 참조데이터가 없어 "수요 카운트"는 되지만
   "수요/용량 = 혼잡도"로 정규화 못 함 → 상대 수요(시간대 상위 %)로 대체하거나 용량기준 확보 필요.
2. **미래/실시간 비행계획 피드 부재**: 현 데이터는 배치·역사. "지금 계획된 이 편이 N시간 뒤 그
   FIR를 지날 때"의 실제 예측은 실시간 비행계획 수신이 있어야 함(= [10 §3](./10-ui-and-realtime.md)
   실시간 결정엔진 로드맵, 미착수). **1단계 대체안**: EOBT + 평균소요로 통과시각을 추정하고 해당
   FIR의 그 **시간대 역사 평균 점유**와 대조(예측 프록시).
3. **경유 FIR별 세분 점유**: flight_data의 entry/exit은 (주로) 국내 FIR 1쌍 → 전 경유 FIR별 세밀
   점유는 CAT62 전처리 또는 `fir_enroute` + 통과시각 추정으로 근사.

### 13.3 AI 추천에 얹는 방법 (단계별)
- **1단계(가벼움)**: `processed_flight_data` entry/exit로 **경유 FIR별 "시간대 혼잡 프로파일"**(통과
  추정시각의 그 시간대 역사 평균 편수 → 상/중/하)을 만들어 프롬프트 맥락에 추가.
  예: "경로2는 ZSHA를 09~10시(역사적 상위 20% 혼잡대)에 통과 예상."
- **2단계(정밀)**: CAT62 전처리 적재 + 섹터 용량 확보 → 진짜 수요/용량 혼잡도, 항로 구간별.
- **3단계**: 실시간 비행계획 수신([10 §3](./10-ui-and-realtime.md)) → 실제 미래 시점 예측.
- ⚠ 이건 **신규 백엔드 작업**(flight_data를 소비하는 `/api/traffic/*` 또는 ODR2 빌드에 시간대 통계
  추가)이라 1단계 AI 프롬프트 MVP와 **분리** 권장. 착수 여부는 §12 D10/D11에서 결정.

## 14. (추가 검토) 실시간 반영 vs 이력 조회 — 데이터 역할 분리

> 사용자 통찰(2026-07-23): 백엔드에 **비행계획서·FOIS·흐름관리가 실시간으로 바뀌면(메시지 수신)
> 그걸 반영해 흐름관리 인지 경로가 추천**돼야 한다. **과거자료는 조회 목적**이지 현재→미래
> 의사결정에 직접 도움이 안 되면 **이력으로 조회하는 구조**가 맞지 않나?

**평가**: 개념 모델은 **맞다.** 그리고 이미 [10 §3.3](./10-ui-and-realtime.md)의 문서화된 목표
("수신만 배치→스트리밍, `processed_*` append-only·최신본 규약 유지, 실시간은 '최근 창' 뷰 추가")와
**정합**한다. 다만 아래 1가지를 정제한다.

### 14.1 데이터 역할 분리 (정제)
"과거 = 이력조회 전용"은 **절반만 맞다.** 두 역할로 나눠야 한다.

| 성격 | 무엇 | 출처(목표) | 결정에서의 쓰임 |
|---|---|---|---|
| **① 현재 결정 사실**(live) | 통과시점에 **발효 중인** 흐름관리 제한, 실제 제출 비행계획, 방금 접수된 FOIS 사유 | 실시간 메시지 수신(미구현) → 현재는 **배치 최신본**을 프록시 | 추천의 **직접 근거** |
| **② 예측 baseline**(history) | 시간대 교통량 프로파일(§13), 재발 흐름관리 패턴, 상습 지연 | 배치 `processed_*` | 통과시점 **혼잡·지연 예측의 사전(prior)** |
| **③ 순수 이력/감사**(history) | 지난 사건 조회·역추적 | 배치 `processed_*` | 결정경로 밖(조회 도구) |

> 즉 과거자료를 결정경로에서 통째로 빼는 게 아니라, **①(직접 사실)은 실시간이라야 하고**,
> ②(예측 baseline)는 여전히 과거자료가 담당한다. ③만 순수 이력조회. 지금의 FOIS/흐름관리
> **수동 조회 패널**(`fois-panel.js`/`flow-management-panel.js`)은 ③에 해당해 그대로 둔다.

### 14.2 소스별 실시간화 그림
- **흐름관리(가장 강한 케이스)**: 제한은 본질적으로 `apply_start_dt`~`apply_end_dt` **유효창**을 가진
  라이브 제약이다. 결정에는 "그 편의 **통과시점에 발효 중인** 제한 집합"이 필요 → 실시간 메시지가
  active 집합을 갱신. 배치 테이블도 이미 유효창을 담으므로 **지금도 "통과시점 발효중" 필터**로
  프록시 가능(§12 D13).
- **비행계획서**: 현재 `processed_flight_data`는 역사 제출분. 실시간 FPL 피드가 오면 **미래 편의 실제
  경로·EOBT**로 §13.2의 "실시간 예측 부재" 간극(#2)을 직접 해소.
- **FOIS**: 실시간 사유 접수 → 현재 진행 중 지연을 결정에 반영(현재는 공항 단위 역사 집계만).

### 14.3 지금 설계에 반영할 원칙 (리워크 방지)
- 추천로직을 **"이력 조회"가 아니라 "현재 결정상태 조회"라는 추상**으로 짠다. 그 추상 뒤를
  **오늘은 배치 최신본**(`latest_run.latest_view` + 유효창 필터)이 채우고, **나중에 실시간 피드**로
  갈아끼운다 — 코드 인터페이스는 불변(§12 D12). 이것이 [10 §3.3](./10-ui-and-realtime.md) "배치→실시간
  전환 원칙"의 구체화다.
- 실제 **실시간 메시지 수신부(이벤트 버스·결정 엔진)는 신규 컴포넌트**라 [10 §3](./10-ui-and-realtime.md)
  대로 **3단계**로 둔다(§12 D14). 이번 1단계 AI 프롬프트 MVP는 이 추상 위에서 **배치 프록시**로 동작.

### 14.4 단계 매핑
| 단계 | 결정 사실(①) 출처 | 이 문서 항목 |
|---|---|---|
| 1단계(지금) | 배치 최신본 + 흐름관리 유효창 필터(프록시) | §5 통합기 · §12 D12/D13 |
| 3단계(로드맵) | 실시간 메시지(FPL·FOIS·흐름관리) 수신 → 결정엔진 | [10 §3](./10-ui-and-realtime.md) · §12 D14 |

- 마스킹 불요: 기상 API는 공개 데이터(민감정보 아님). CORS는 기존 프록시 체인/기상서버 화이트리스트로 해결.

  종합하는 것이 핵심(§12 D17).

## 16. (추가 검토) 공항 운항(활주로·스탠드) 반영 — 터미널 결정 계층

> 사용자 통찰(2026-07-23): 공항 운항도 함께 고려 — **현재 어떤 활주로가 붐비니까 다른 활주로·다른
> 지점으로 가면 더 효과적**이라는 것도 같이 의사결정되면 좋겠다.

**평가**: 타당하고 **원자료가 있다.** 다만 이건 en-route(OD 경로) 추천과는 **다른 결정 계층
(터미널/공항)** 이라, 범위를 명확히 나눠 다룬다.

### 16.1 가용 자산 (공항 운항)
- **ACDM**(`processed_acdm_departure`/`arrival`): **`runway`·`stand`·`gate` + 시각**(dep: SOBT/TOBT/
  TTOT/CTOT/AOBT/ATOT, arr: ELDT/ALDT/SIBT/AIBT) + `taxi_out_additional_min`(혼잡 신호)·정시성·CTOT준수.
  → **활주로별 시간대 수요·택시지연·정시성**을 뽑을 원자료가 다 있다.
- **기구현 엔드포인트 `GET /api/airports/{icao}/ops`**(`backend/app/queries/airport_ops.py`): ⚠ 단
  **공항 단위 집계만**(정시율·평균택시·CTOT준수, dep/arr 분리) — **활주로별 group by·시간대 버킷은
  아직 없음**(메모리에는 "2단계 이관"으로 남아 있으나 실제로는 공항 단위까지 구현·등록됨). 프론트도
  동시세션이 독립 패널 `ops-panel.js`(uncommitted) + `adapters.toAirportOpsSummary`(`on_time_rate` 등)로
  붙이는 중 — 스크린샷의 "일평균/정시율/택시아웃" 타일이 이것.
- **SID/STAR 참조**(`GET /reference/sidstar`, `adapters.toSidStar`): 활주로 선택 → 출발/도착 절차 →
  **초기/최종 경로 지점**의 연결고리. "다른 활주로 = 다른 SID = 다른 진입점"을 지오메트리로 연결.

### 16.2 격차 (신규 필요)
- **활주로/스탠드별 + 시간대 버킷 수요 집계**(어느 활주로가 언제 붐비나)는 **미구현** → 기존
  `airport_ops.py`에 `group_by(runway, time_bucket)` 집계를 **증분 확장**하면 됨(신규 대공사 아님).

### 16.3 결정 계층 구분 (중요)
- 현 ODR2 추천 = **en-route**(FIR·순항 경로 옵션). 활주로·SID 선택 = **터미널/공항** 계층.
  둘은 **SID/STAR로 연결**되지만 결정 granularity가 다르다.
- 따라서 "활주로 혼잡 → 다른 지점 경로"는 **지상~순항~지상을 아우르는 통합 결정(gate-to-gate)** 으로 범위가 넓어진다.
  AI 프롬프트에 **두 맥락(en-route + 공항운항)을 함께** 주면 종합 판단은 가능하나, 1단계는 en-route
  중심으로 두고 공항운항은 **참고 KPI**로 얹는 선에서 시작 권장(§12 D18/D20).

### 16.4 제약 (오해 방지)
- **역사 → 실시간(§14 연계)**: "지금 어느 활주로가 붐비나"의 *직접 사실*은 실시간 ACDM 배정이라야
  하고, 현재는 **활주로 시간대 수요 프로파일**(역사 baseline)로 프록시. 같은 배치→실시간 축.
- **활주로 배정은 자유선택 아님(§15 연계)**: active 활주로 config는 보통 **바람 방향**(정풍 이착륙)으로
  결정된다. "덜 붐비는 활주로로 가라"는 추천은 **현 활주로 config·바람 제약을 존중**해야 하며, ATC
  통제 사항임을 프롬프트/팝업에 명시(§9 자문 톤과 동일).

### 16.5 단계 매핑
| 단계 | 공항운항 프롬프트 | 신규 작업 | 항목 |
|---|---|---|---|
| 1단계(지금) | 출/도착 공항 **KPI**(기구현 `/ops`: 정시·택시·CTOT) | 없음(호출만) | §12 D18 |
| 후속 | **활주로별 시간대 수요** + SID/STAR 연계 대안지점 | airport_ops 확장 + 터미널 결정 통합 | §12 D19/D20 |

## 17. (추가 검토) "왜 이 경로 + 막히는 구간 예측" — AI 출력의 핵심 + 구체정보 표시

> 사용자 통찰(2026-07-23, 2번째 스크린샷=완성본 원본): 경로 추천 시 **이미지처럼 구체적 정보를 함께
> 표시**하면 유리하고, **왜 이 경로를 써야 하는지 + 막히는 구간이 있는지**가 예측되면 좋겠다.

**평가**: 이 요청이 이 문서 전체의 **목표 출력을 확정**해 준다 — 세 부분으로 나뉜다.

### 17.1 "구체적 정보 표시" = 대부분 완성본이 이미 함 → advisor 포팅 로드맵(AI 신규작업 아님)
- 스크린샷(완성본 원본)이 경로별로 이미 보여주는 것: 정시성·지연%, 흐름관리%, 실적 고도 분포,
  경유/인천 FIR 픽스체인, ops KPI(일평균·정시율·택시아웃), 경로 상층풍·난류, 강수 레이더, ★권장/최단 비교.
- ⚠ 이건 **AI 신규 작업이 아니라 프론트 포팅(docs 04) 로드맵**이다 — advisor는 지금 일부만 포팅됨
  (ops 어댑터·`radar.js`·`hazards.js` 동시세션 진행, 경로별 정시성/흐름관리%/상층풍 브리핑은 미포팅).
- **AI 계층 원칙**: 이 값들을 **재계산하지 말고 "이미 계산·표시된 값을 프롬프트 입력으로 소비"**
  한다(§5 통합기 강화, §12 D22).

### 17.2 "왜 이 경로" = AI 근거 (이미 §6/§8이 다룸)
- §8-2(결정론적 점수 + AI 근거) 그대로. 위 구체정보를 종합해 **자연어 2~3문장 "왜"**를 생성 → 팝업(§N4).

### 17.3 "막히는 구간 예측" = 세그먼트 단위 병목 — 이 문서 축들의 종합 출력 (신규)
- **정의**: 경로를 FIR/세그먼트로 쪼개 각 구간의 **막힘(병목) 가능성·원인**을 예측·표시:
  - **교통량 병목**(§13): 통과추정시각에 그 구간의 역사 평균 점유(상/중/하).
  - **흐름관리 병목**(§14): 통과시점 발효 중인 MIT/MINIT/rate 제한이 걸린 FIR/픽스.
  - **기상 병목**(§15): 그 구간의 상층풍·시어·SIGMET·강수셀 교차(스크린샷의 빨강 강수역이 경로 횡단).
- **출력 형태**: "경로2의 ZSHA 구간 09~10시 혼잡 상위 + 강수셀 교차 → 지연 위험"처럼 **구간별 근거**를
  프롬프트에 넣고 AI가 요약, 지도에는 해당 구간 강조(§15 ④ 루트 교차 경고와 결합).
- ⚠ **단계 현실**: 완전한 세그먼트 병목은 §13(교통)·§15①(상층풍)·④(루트교차) 선행 필요 → **후속**.
  **1단계는 이미 가용한 구간 신호**(경로별 흐름관리% · 경유 FIR별 흐름관리 유효창 · 공항 FOIS/기상)로
  **부분 병목 힌트**부터 프롬프트에 넣어 시작(§12 D21).

### 17.4 정리
| 사용자 요청 | 정체 | 담당 | 단계 |
|---|---|---|---|
| 구체정보 표시(§17.1) | 완성본이 이미 함 | **프론트 포팅**(docs 04), AI는 소비 | 포팅 로드맵 |
| 왜 이 경로(§17.2) | AI 근거 | AI(§6/§8) | 1단계 |
| 막히는 구간(§17.3) | 세그먼트 병목 종합 | 흐름관리+공항(1단계) → 교통+기상+교차(후속) | 부분→완전 |

## 18. (추가 검토, v1.1) SUAS/MOA 시간대별 발효 — §17.3 "막히는 구간"의 신규 신호

> 사용자 통찰(2026-07-23): 경로 추천 시 MOA(군작전구역) 등 특수공역은 **평일엔 사용 불가, 주말엔
> 사용해야 하는 경우**가 있고, 실제 비행계획서는 이를 고려해 제출된다. 일본 쪽에도 유사한 군사공역이
> 있는지 확인 요청.

### 18.1 확인 결과 — 데이터는 있으나 파이프라인에서 누락됨
- **원본에 존재**: DAFIF `SUAS_PAR.TXT`(`PORTING_PACKAGE_ROOT/원본데이터/DAFIFT/SUAS/`, §0.1 읽기전용
  외부자산)에 **`EFF_TIMES`(발효시간) 컬럼**이 있다. 전체 18,426건 중 **16,327건에 실값**.
  - 한국(RK) 예: `ASAN`/`HANAM`/`BUCHEON` → `MON-FRI 2300-1300Z, SAT 2300-0400Z`(평일 장시간, 토요일
    단축, 일요일 미기재) — 사용자가 말한 "평일/주말 다른 사용" 패턴이 텍스트로 그대로 있음.
  - **일본(RJ)도 확인됨**: SUAS 총 **264건**(A 117·**M(MOA) 66**·R 30·T 40·W 11), 이와쿠니(IWAKUNI)
    미군기지 훈련구역 다수 포함. 요일 패턴 예: `MON-FRI 2200-1200Z, OT BY NOTAM`, `MON-SAT ...`.
- **파이프라인 누락**: 빌드된 `사전빌드_JSON/suas.json`/`suas_world.json`의 실제 스키마는
  `[ident, name, type, upper_alt, lower_alt, polygon]` — **`EFF_TIMES`가 빠져 있다**. 빌드 스크립트
  (`전처리스크립트/5_build_map3.py`)는 이미 만들어진 suas.json을 불러오기만 할 뿐, `SUAS_PAR.TXT →
  suas.json` 변환 자체에서 이 필드를 다루는 코드가 없다(의도적 제외가 아니라 **누락**으로 판단,
  `문서/01_데이터명세.md`에도 언급 없음).
- **비행자료(`비행자료검색_*.xlsx`)에는 명시적 회피 사유 필드 없음** — `ROUTE`/`FIXPNT1~30`으로 실제
  경로만 있고, MOA 회피가 암묵적으로 반영돼 있을 뿐 "왜 이 경로"에 대한 명시 컬럼은 없다.
- **또 다른 선행 갭**: SUAS 참조 레이어 자체가 advisor에 **아직 미구현**이다([03 §4](./03-backend-api.md)
  `/api/reference/suas`는 "(2단계)"로만 명세돼 있고 실제 라우터/쿼리 코드 없음, [04 §7](./04-frontend-migration.md)
  "지연 로드: sidstar/suas(2단계)"). 즉 이 신호를 쓰려면 **(a) SUAS 참조 레이어 포팅 + (b) EFF_TIMES
  보존·파싱**의 2단 작업이 필요하다.

### 18.2 EFF_TIMES 파싱 난이도 — 구조화 vs 비정형
자유텍스트라 전부 신뢰성 있게 파싱할 수는 없다. 두 부류로 나뉜다.

| 부류 | 패턴 예 | 파싱 가능성 |
|---|---|---|
| **구조화 가능** | `MON-FRI`/`MON-SAT`/`SAT`/`SUN`/`DLY` + `HHMM-HHMMZ` UTC 시간범위 | 규칙 기반 파싱 가능(요일 키워드 + 시간범위 정규식) |
| **비정형(파싱 위험)** | `SR-SS`(일출-일몰, 위치·계절 따라 변동) · `BY NOTAM`/`OT BY NOTAM` · 자유서술 각주 | **억측 금지** — 발효 여부를 함부로 단정하면 안전 문제. "확인 필요"로만 표시 |

### 18.3 결론 — §17.3 "막히는 구간"에 신규 하위신호로 추가(D23)
- 경로가 통과하는 SUAS/MOA 중, **통과 예상 시각에 구조화 패턴상 발효 중**인 것을 병목/우회 근거로
  제시한다(예: "경로 X는 화~금 09시 MOA 활성 시간대 통과 — 우회 가능성 확인"). 비정형 패턴은 발효를
  단정하지 않고 "NOTAM 확인 필요"로만 노출.
- 이건 여전히 **결정론 신호**(AI 아님, Phase A) — AI는 이 판정 결과를 서술만 한다(§17.1 원칙과 동일).
- 구현 스펙은 [13 STEP A7](./13-ai-reasoning-dev-plan.md).
