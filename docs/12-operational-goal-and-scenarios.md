## 1. 근본 목적 (재정의)

이 서비스의 원래 목적은 **"OD 간 순항 경로(en-route)를 고르는 것"에 그치지 않는다.** 실제 동기는
**혼잡·제약 상황에서 지상/공중 대기를 줄이도록 출발·도착의 진출입 지점과 활주로, 그리고 흐름관리
파라미터를 종합 조정**하는 것이다. 구체적으로:

1. **출발측**: 인천공항 활주로 혼잡 시, **활주로에서 계속 기다리는 것보다 다른 활주로로 출발해
   진출지점을 바꿔 나가는 것이 더 효과적**일 수 있다(특히 일본행).
2. **도착측**: 도착지 커퓨(운항금지시간) 등으로 **공중 대기(holding)** 가 생길 때, **진입 지점을 바꿔
   공중 대기 시간을 줄인다**(예: 대만→김해).
3. **흐름관리**: 위 판단과 함께 **항공기 출발시각·도착 분리간격(separation) 조정**을 지원한다.
4. 이 모든 것을 **자동화**해 의사결정을 돕는다.

> 즉 핵심 결정 계층은 **터미널(활주로·진출/진입 픽스) + 흐름 파라미터**다. 현 ODR2(순항 경로 옵션)는
> 이 결정을 **감싸는 맥락**이지 목적 그 자체가 아니다. → [11 §16 D20](./11-ai-route-reasoning-proposal.md)의
> "en-route 우선, terminal 후속" 기본값은 이 목적에 비추어 **재검토 대상**(§7).

## 2. 핵심 의사결정 시나리오

### S1. 인천 출발 — 활주로 혼잡 → 진출점 전환 (특히 일본행)
- 인천공항 **활주로 4개**, 각 활주로가 특정 **출발 방향(진출 픽스)** 과 연결된다.
- 한쪽 활주로/진출 방향이 계속 막히면, **다른 활주로로 출발해 진출지점을 바꾸는** 편이 **활주로에서
  대기(지상대기)** 하는 것보다 총소요가 짧을 수 있다.
- **판단 구조**: `지상대기(현 활주로) 예상` vs `활주로 전환 + 대체 진출점 재라우팅 비용`. 후자가 작으면 전환 추천.

### S2. 중국 vs 일본 — 진출 자유도의 비대칭 (제약 규칙)
- **중국행**: 전날에 우리나라에서 중국으로 나가는(그리고 우리나라로 들어오는) 항공기가 **이미 확정**된다
  → 진출 지점/슬롯이 **경직**(임의 변경 어려움).
- **일본행**: 진출 방향이 **상대적으로 자유** → 진출 지점이 **동적으로 바뀔 수 있다**(예: **LANAT → BESNA**).
- **함의**: S1의 "진출점 전환"은 **일본행에서 실질적 옵션**이고, 중국행은 제약이 커 옵션이 제한된다.
  추천 엔진은 이 **목적지별 제약 비대칭을 규칙으로** 반영해야 한다(중국행에 자유로운 전환을 제안하면 오답).

### S3. 대만 → 김해 새벽 도착 — 커퓨 → 진입점 전환으로 공중대기 저감
- 대만발이 **새벽에 김해 도착** 시 **커퓨(운항금지시간)** 로 **공중 대기(holding)** 가 발생.
- 통상 **ATOTI**로 진입하는데, **RUGMA**로 진입하면 **공중 대기 시간이 다소 줄어드는** 것으로 관측됨.
- **판단 구조**: 커퓨 개시/해제 시각 대비 도착 예정시각을 보고, **진입 픽스 대안(ATOTI↔RUGMA)** 중
  공중대기가 최소인 것을 추천.

### S0(종합). 흐름관리 파라미터 조정
- 위 판단과 맞물려 **출발시각(슬롯) 조정**·**도착 항공기 분리간격(MIT/MINIT) 조정**을 함께 제안 →
  섹터/활주로 수요를 평활화.

## 3. 데이터 가용성 — 시나리오별 매핑

| 필요 정보 | 소스/컬럼 | 상태 |
|---|---|---|
| 활주로(어느 활주로로 나갔나) | ACDM `runway`, `stand`, `gate`(schema §9.2/9.3) | ✅ 있음 |
| 진출/진입 픽스 체인(노선별) | ODR2 옵션 `incheon_track_fixes`(routes.py), 스크린샷 "진입 BOPTA→진출 SADLI" | ✅ 있음 |
| 개별 픽스 좌표(LANAT·BESNA·ATOTI·RUGMA) | `/reference/waypoints` | ⚠ **존재 추정, 이번 세션 권한이슈로 미확인** → 확인 필요 |
| **활주로 → 진출점 매핑** | SID/STAR(`/reference/sidstar`)에 **runway 필드 없음**(`toSidStar={proc,name,airport,coords}`) | ⚠ **갭** — SID명 인코딩 파싱 or **ACDM.runway × 노선 진출픽스 조인** 필요 |
| 출발시각 관리(슬롯) | ACDM `sobt/tobt/ttot/ctot/aobt/atot`, `taxi_out_additional_min` | ✅ 있음 |
| 도착시각·공중대기 근사 | ACDM `eldt/aldt/sibt/aibt`, `fir_to_app_min`, `final_approach_min` | ✅ 있음(홀딩 직접값은 근사) |
| **커퓨(김해 등) 시간대** | docs/데이터에 **참조 없음** | ⚠ **갭** — 공항별 커퓨 **정적 참조 신규 필요** |
| 분리간격 | flow_management `mit`, `minit`, `alt_speed_limit` | ✅ 있음 |
| 진입/진출 실제 시각 | flight_data `entry_fir/entry_datetime`, `exit_fir/exit_datetime`, `full_route` | ✅ 있음 |
| 시간대 수요/혼잡(활주로·픽스) | flight_data·ACDM 시간대 집계 | ⚠ 신규 집계 필요([11 §13·§16](./11-ai-route-reasoning-proposal.md)) |

> **가장 큰 두 갭**: (a) **활주로→진출점 매핑**(SID에 runway 없음 → 조인/파싱 설계 필요), (b) **커퓨
> 참조 데이터 부재**(신규 정적 참조). 이 둘이 S1·S3의 핵심 신호원이라 **선행 확보 대상**이다.

## 4. 자동화 대상 (무엇을 자동 판단/추천하나)

| 결정 | 입력 | 출력 | 시나리오 |
|---|---|---|---|
| **출발 활주로·진출점** | 활주로별 혼잡(지상대기) + 대체 진출점 재라우팅 비용 + 목적지 제약(§2) | "활주로 X 지상대기 큼 → 활주로 Y+진출점 BESNA 전환 권장(총 −N분)" | S1·S2 |
| **도착 진입점** | 커퓨 시각 vs 도착 예정 + 진입점별 공중대기 | "커퓨로 홀딩 예상 → RUGMA 진입 시 공중대기 −N분" | S3 |
| **흐름 파라미터** | 섹터/활주로 시간대 수요 + 발효 흐름관리 | "출발슬롯 +N분 / 도착 분리간격 MIT 조정 제안" | S0 |

- 이 출력들은 [11 §17](./11-ai-route-reasoning-proposal.md)의 "왜 이 경로 + 막히는 구간"과 결합된다 —
  **결정론적 계산(tradeoff·제약)** 이 후보를 만들고, **AI가 근거를 자연어로 서술**([11 §8-2](./11-ai-route-reasoning-proposal.md)).

## 5. 제약·주의 (오해·오답 방지)

- **목적지 제약 비대칭(§2)은 하드 규칙**: 중국행에 자유 진출점 전환을 제안하면 오답. 규칙으로 명시.
- **커퓨는 규정·안전 하드 제약**: 커퓨 창을 "참고"가 아니라 **불가 조건**으로 다룬다.
- **활주로 배정·진출입 지정은 ATC 통제**: 추천은 **자문(advisory)**, 최종판단은 관제/사용자
  ([11 §9](./11-ai-route-reasoning-proposal.md) 톤과 동일).
- **관측 기반 가설 표기**: "RUGMA가 공중대기 짧다"는 사용자 관측 → 데이터로 **검증 후** 근거화(발명 금지).
- **역사 vs 실시간**([11 §14](./11-ai-route-reasoning-proposal.md)): "지금 어느 활주로/섹터가 막혔나"의
  직접 사실은 실시간이라야 하고, 현재는 배치 최신본 + 시간대 프로파일로 프록시.
