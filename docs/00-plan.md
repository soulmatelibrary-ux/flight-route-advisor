## 1. 배경

`비행경로추천서비스_이식패키지`는 전세계 항공로·공항·FIR과 인천 FIR 실운항 기반 경로추천을 보여주는 인터랙티브 지도다. 현재 구조는 다음과 같다.

- 모든 데이터(항공로 89,555 / 공항 10,030 / 픽스 58,812 / FIR 247 / 경로추천 ODR2 등)를 **빌드타임에 JSON으로 굳혀 단일 HTML(~15MB)에 `const`로 임베드**한다.
- 런타임 DB·API가 없다. 데이터가 바뀌면 전처리 스크립트로 **HTML 전체를 재조립**해야 한다.
- 유일한 런타임 서버는 Node 기상 프록시(`aviation-weather-mcp`)뿐이다.

이 방식은 단일 파일 배포에는 유리하지만, 실운항 데이터가 주기적으로 갱신되고(월 단위 비행자료·일 단위 ACDM/FOIS) 조회·필터가 필요한 **서비스**로 키우기에는 한계가 있다.

이를 위해 실운항 원본(비행자료·ACDM·FOIS·흐름관리일지)을 전처리해 **PostgreSQL에 적재하는 백엔드**(설계: `data-ingestion-backend/docs/`)를 **Stage 0로 이 저장소에서 먼저 개발**한다. 그 DB가 채워지면 지도 서비스(Stage 1·2)는 임베드된 정적 스냅샷 대신 **DB의 최신 운항 데이터를 동적으로** 소비한다. 순차 개발 순서·환경은 [08-setup-and-dev-order](./08-setup-and-dev-order.md), 진행 추적은 [07-checklist](./07-checklist.md).

## 2. 목표

**비행경로추천서비스를 "정적 임베드"에서 "DB 기반 동적 웹서비스"로 전환하는 설계를 확정한다.**

- 자주 바뀌는 운항·분석 데이터(경로추천·ACDM 정시성·FOIS 지연·흐름관리)는 **전처리 DB → FastAPI → 프론트 fetch**로 흐르게 한다.
- 잘 안 변하는 참조 지오메트리(DAFIF 항공로/공항/FIR/픽스)는 **캐시된 벌크·타일 엔드포인트**로 서빙해 초기 로딩 성능을 지킨다(구분 서빙).
- 기상·실시간 항공기(ADS-B) 등 외부 API 연동과 렌더링 알고리즘은 원본 설계(`문서/03·04·07`)를 보존한다.

이번 단계 산출물은 **설계 문서**이며 코드는 만들지 않는다.

## 3. 범위 (Scope)

### In Scope (이번 설계에 포함)
- 정적 임베드 → 동적 fetch 전환 아키텍처
- 전처리 DB(`processed_*` 테이블) 연동 방법 (연결·질의·집계·어댑터·Supabase 이전) — **사용자 강조 항목**
- FastAPI 읽기 전용 조회 API 명세 (MVP 엔드포인트)
- 참조 지오메트리 구분 서빙 전략
- MVP 범위 정의와 향후 확장 로드맵
- 이식 규약·함정 정리

### Out of Scope (이번 설계 제외 / 향후)
- 실제 코드 구현·배포
- 경로 기하 동적 재계산(buildGeo, `문서/04-D`) — 사전집계로 대체
- 상층풍·시어 오버레이 동적화(`문서/04-E`)
- 실시간 ADS-B 충돌예측(STCA/CPA)
- 흐름관리 탭·FIR 분석 패널 (전처리 통합데이터 의존)
- 통합데이터(84,520×122)·흐름관리 영향상세 기반 기능 (전처리 측 통합 스킬 미완성)
- **전처리 스킬 로직 자체**(수정 없이 재사용) — 별도 `데이터전처리기술이식` 프로젝트 소관. ⚠ 이는 그 로직을 호출하는 **Stage 0 적재 백엔드 앱**(`flight-route-advisor/data-ingestion-backend/app·docker`, §1·§4)과는 다른 대상이다 — Stage 0 앱 자체는 **이 저장소 범위 안**([08 §2.1](./08-setup-and-dev-order.md)).

## 4. 3단계 로드맵

| 단계 | 이름 | 핵심 산출물 | 완료(검증) 기준 |
|---|---|---|---|
| **1** | **MVP — 동적 기반 구축** | 참조 지도(FIR/항공로/공항/픽스/항행시설/TCA) + 공항 기상 팝업 + 경로추천 조회(사전집계 ODR2) + ADS-B 실시간 | 정적 HTML과 동일한 지도가 fetch 기반으로 뜨고, `dep/arr` 선택 시 DB 유래 경로추천이 표시됨 |
| **2** | **경로추천 심화** | 경로 기하 동적 재계산(04-D), 상층풍·시어 오버레이(04-E), ACDM 정시성/FOIS 지연 패널 | 옵션 선택 시 FIR 경유 면·시어 색 오버레이가 원본 완성본과 시각적으로 일치 |
| **3** | **실시간·분석** | 실시간 ADS-B STCA/CPA, 흐름관리 탭, FIR 분석 패널 | 통합데이터 테이블 확보 후 흐름관리 영향·섹터 수요예측 표시 |
| **향후** | **실시간 의사결정 지원** | 비행계획서·ACDM·FOIS 실시간 수신→스트림 처리→"지금 결정 큐"→추천/알림 ([10](./10-ui-and-realtime.md) §3) | 이벤트 버스·결정 엔진 신규. 현 MVP는 배치 유지, 별도 설계·검증 단계 |

> UI 방향: 전세계 지도를 보유하되 **의사결정 중심(선택 FIR·루트만 표시)**으로 전환 — [10-ui-and-realtime](./10-ui-and-realtime.md).

각 단계는 이전 단계의 검증을 통과한 뒤 착수한다. 상세 범위는 [05-mvp-scope](./05-mvp-scope.md).

## 5. 의존성

1. **전처리 DB 스키마** — `processed_*` 테이블 물리 컬럼 매핑은 [DB스키마 §9](../data-ingestion-backend/docs/DB스키마.md)로 **확정됨**([02-db-integration](./02-db-integration.md) §5 반영). 잔여 의존은 아래 #2(통합데이터 테이블)뿐. 헤더 변경 시에만 `column_map.py` 재동기화.
2. **통합 스킬/통합데이터 테이블** — 흐름관리 영향상세·통합데이터(84,520×122)는 전처리 1단계에서 제외됨. 3단계 기능의 선행조건.
3. **기상서버** — 기존 Node `aviation-weather-mcp`가 켜져 있어야 공항 기상·CORS 프록시가 동작(`문서/07`).
4. **참조 지오메트리** — DAFIF 기반 사전빌드 JSON(`비행경로추천서비스_이식패키지/사전빌드_JSON`) 또는 재파싱본.

| 실운항 데이터(비행계획) 민감성 | 정보 취급 주의 | 서버사이드 읽기 전용 접근, 필요 시 집계 단위로만 노출 |
